### B. Observer execution model (finding 3) — the core redesign
Make observation genuinely unable to affect (a) the outcome value, (b) when the transport gets it,
or (c) other observers.
1. `DispatchAsync` computes the outcome and **returns it without awaiting observers**. Observer
   notification is started and surfaced as a new `DispatchOutcome.ObserverNotification` (`Task`,
   default `Task.CompletedTask`). Transports ignore it → reply delivery is never gated. Tests await it
   for determinism.
2. Each observer is invoked **independently** (its own guarded task), so a never-completing observer
   cannot starve the others; results aggregated into `ObserverNotification`.
3. Remove the cancellation rethrow from the notify path — an observer's cancellation/fault is logged
   and confined to `ObserverNotification`; it can never propagate into the returned outcome.
4. XML/PRD/CHANGELOG: state the real contract — observers run **off the reply path**, **may be invoked
   concurrently** (across messages and relative to each other), **once per matching message**, and
   **cannot affect the outcome**. Drop the "sequential" implication.
5. Tests: never-completing observer → outcome+reply returned immediately, DF waiter still completes;
   cancellation during an observer → outcome unchanged, other observers still run; concurrent dispatch
   into a stateful observer; slow observer does not delay WS same-socket reply (interop).

*Open design question for the user (below): the `ObserverNotification`-Task approach vs. an internal
bounded background queue. I recommend the Task approach — smallest change, fully testable, no new
infra — but it does make observers fire-and-forget from the transport's view.*
