## Deviations from the approved plan

1. **`UnreservedUriChars.cs` added** — the plan didn't enumerate this file but `Message`,
   `Attachment`, and the `thid`/`pthid` checks all needed a shared RFC 3986 §2.3
   classifier. ≈30 LOC, internal, single responsibility.
2. **`Message.Body` `null` ↔ absent** — the plan said "default to empty `{}` when
   sending" for v2.0 compatibility. The model itself leaves `body` nullable; the
   builder leaves it `null`; the `WhenWritingNull` ignore policy keeps it off the wire
   when unset. The "emit `{}` on send for v2.0 compat" decision was reclassified as
   **packing-layer responsibility** (Phase 2 envelope construction will set
   `body ??= new JsonObject()` immediately before serializing). Phase 1's model stays
   round-trip-lossless: a wire message with no `body` survives unpack→repack with no
   `body` rather than gaining a spurious `{}`.
3. **`Appendix_C1_round_trips_structurally`** compares field-by-field plus
   `DeterministicJsonWriter.WriteString(body)` rather than `BeEquivalentTo` on the
   whole `Message`. FluentAssertions' equivalence walker trips on `JsonNode`'s
   `Parent`/`Root` self-references and reports a cyclic-reference error; the
   field-by-field assertion is equivalent and avoids the walker.

## Open spec/PRD inconsistency to flag

- **FR-MSG-02 "≤ 32 bytes" vs. FR-MSG-03 "UUID v4 default"** — UUID v4 string form is
  36 characters, which exceeds the 32-byte cap claimed by FR-MSG-02. The current code
  enforces the unreserved-character rule (the security-relevant part: ids compose URIs)
  and **omits the 32-byte length cap** so the default generator's output is accepted by
  `Validate()`. The DIDComm v2.1 spec itself does not specify a length cap; the PRD
  text appears to be in error. Suggest the PRD be revised to either drop the "≤ 32
  bytes" phrase or raise the cap to ≥ 36 bytes. Filed here for the next PRD-sync pass.
