# Refactor: adopt NetCrypto v1.0.0 as the crypto substrate; delete hand-rolled crypto; keep DIDComm JOSE composition

**Date:** 2026-06-14
**Status:** PLAN — awaiting approval (no source/test/doc edits made yet)
**Driver:** `architectural-path.md` — separate concerns so each subsystem is lean. didcomm-dotnet should consume the foundation crypto (`crypto-dotnet` = **NetCrypto**) and proofs (`dataproofs-dotnet` = **DataProofsDotnet**) rather than carrying its own.

---

## 1. Executive summary (what the analysis found)

A 18-agent workflow inventoried both foundation packages and didcomm's `Crypto/` + `Jose/` layers, cross-referenced them, and adversarially verified every replace/gap claim against the **shipped binaries**.

1. **NetCrypto v1.0.0 is a byte-compatible drop-in for didcomm's entire primitive layer.** It ships `AesCbcHmacCipher` (A256CBC-HS512), `AesGcmCipher` (A256GCM), `XChaCha20Poly1305Cipher` (XC20P), `AesKeyWrap` (A256KW, RFC 3394), `ConcatKdf` (ECDH-ES **and** the ECDH-1PU two-secret `Ze‖Zs` shape with tag-in-suppPubInfo), `ICryptoProvider`/`DefaultCryptoProvider` (raw ECDH + sign/verify with `EcdsaSignatureFormat.IeeeP1363`), `EcPointValidator`, `JwkConverter`, `DefaultKeyGenerator`, `KeyType`, `KeyTypeExtensions.FromMulticodec`. **P-521 ECDH + ES512 are confirmed implemented in the shipped DLL** (refuted the "P-521 might be missing" worry via a runtime test).

2. **DataProofsDotnet contributes nothing to didcomm — do NOT add it as a dependency.** The preview (`0.1.0-preview.1`) ships **only** W3C Data Integrity embedded proofs (`DataProofsDotnet.DataIntegrity.*`: cryptosuites, pipelines). It has **zero JOSE/JWS/JWE/JWK**. DIDComm v2.1 secures messages exclusively with JOSE and explicitly does **not** use Data Integrity, so the package is entirely off-path. (The task framing "remove data proofs already implemented in DataproofsDotnet" has no target: didcomm has no Data-Integrity code; its "proofs" are JWS/JWE, which DataProofs does not cover.)

3. **A trivial NetDid version bump comes along for the ride (not a blocker).** didcomm currently pins **NetDid.Core 1.3.0**, which *bundles* the crypto primitives. **NetDid.Core 2.0.0 is already published** — it removed all crypto and now `depends on NetCrypto 1.0.0 + NetCid 1.6.0` (exactly the architectural-path split). So the move is a one-line bump: **NetDid.Core 1.3.0 → 2.0.0 (resolution-only) + add NetCrypto 1.0.0 (crypto)**. APIs verified byte-identical across the split, so re-pointing is mechanical — no upstream coordination needed.

4. **The whole envelope-composition layer stays in didcomm — this is the one real GAP.** `JweBuilder/JweParser`, `JwsBuilder/JwsParser`, the JOSE header DTOs, `ApuComputer/ApvComputer`, `EnvelopeDetector`, `JoseAlgorithms`, the `Jwk` model and `Base64Url` have **no foundation home** (NetCrypto is primitives-only; DataProofs is Data-Integrity-only). Per PRD §3.1/§12 the JOSE composition is DIDComm-owned. These already delegate every raw primitive downward and are correct as a thin composition shell.

5. **Blast radius is small.** Every non-`Crypto`/non-`Jose` consumer (EnvelopeReader/Writer, ForwardWrapper, ForwardProcessor, FromPrior builder/validator, DidCommClient, DI extensions) depends only on didcomm's **own** `DefaultCryptoProvider` facade. As long as that facade's name/surface is preserved, those files need **zero** edits.

**Net result:** ~**435 LOC** of hand-rolled, security-sensitive crypto deleted (AesKeyWrap 150 + AesCbcHmacSha512 132 + AesGcmAead 74 + XChaCha20Poly1305Aead 79), the **direct** `NSec.Cryptography` reference dropped, one crypto substrate (`NetCrypto`) instead of `NetDid.Core(+bundled crypto) + NSec`, and `NetDid.Core` demoted to a resolution-only dependency — with all DIDComm-specific orchestration intact.

## 2. Disposition table (per component)

### DELETE (delegate to NetCrypto) — ~435 LOC of hand-rolled crypto
| didcomm file | NetCrypto replacement | Notes |
|---|---|---|
| `Crypto/KeyWrap/AesKeyWrap.cs` (150) | `NetCrypto.AesKeyWrap.Wrap/Unwrap` | Highest-value: deletes a hand-rolled RFC 3394 impl + audit burden. |
| `Crypto/Aead/AesCbcHmacSha512.cs` (132) | `NetCrypto.AesCbcHmacCipher` | A256CBC-HS512 (authcrypt enc). Byte-identical construction. |
| `Crypto/Aead/AesGcmAead.cs` (74) | `NetCrypto.AesGcmCipher` | Both thin BCL `AesGcm` pass-throughs. |
| `Crypto/Aead/XChaCha20Poly1305Aead.cs` (79) | `NetCrypto.XChaCha20Poly1305Cipher` | Removes the only direct `NSec` consumer + manual `[ct‖tag]` split. |

### RE-POINT (namespace swap; stays as DIDComm composition)
| didcomm file | Change |
|---|---|
| `Crypto/DefaultCryptoProvider.cs` | Wrapped field/ctor `NetDid.Core.Crypto.DefaultCryptoProvider` → `NetCrypto.DefaultCryptoProvider`; `AeadEncrypt/Decrypt`→NetCrypto ciphers; `KeyWrap/Unwrap`→`NetCrypto.AesKeyWrap`; rename internal `NetDidProvider` accessor. **Keep the public type name + ctor surface** (blast-radius firewall). |
| `Crypto/Kdf/EcdhEsKdf.cs`, `Crypto/Kdf/Ecdh1PuKdf.cs` | Swap `using` aliases for `ConcatKdf` + raw-ECDH to NetCrypto. **Keep all orchestration** (single-Z vs `Ze‖Zs` order, suppPubInfo tag-framing per FR-ENC-15). |
| `Crypto/KeyAgreement/EphemeralKeyPair.cs` | `DefaultKeyGenerator`/`IKeyGenerator`/`KeyType` → NetCrypto. Keep epk-JWK build + `Clear()`. |
| `Crypto/KeyAgreement/KeyTypeMapper.cs` | `KeyType` enum → `NetCrypto.KeyType`. (Keeps the crv/enc→size + JWS-alg tables — no foundation surface for these.) |
| `Jose/JwkConversion.cs` | `JwkConverter` + `KeyType` → NetCrypto. **Keep the OKP-32-byte guard.** **Add an explicit `EcPointValidator.EnsureOnCurve` call** (see Gap G2). |
| `Jose/Encryption/JweBuilder.cs`, `JweParser.cs`, `Jose/Signing/JwsBuilder.cs`, `JwsParser.cs` | Drop the direct `NetDid.Core.Jwk.JwkConverter` alias; route key extraction through didcomm's own `JwkConversion.ExtractPublicKey`. Re-point the `cryptoProvider.NetDidProvider` seam. Composition logic unchanged. |
| `Resolution/NetDidKeyService.cs` | Crypto calls → NetCrypto (`JwkConverter.ToPublicJwk`, `KeyTypeExtensions.FromMulticodec` — note rename from `ToKeyType`). **Keep** NetDid.Core resolution types + NetCid Multibase/Multicodec. |
| `Adapters.NetDid/NetDidKeyStoreSecretsResolver.cs` | `IKeyStore`/`StoredKeyInfo`/`JwkConverter` → NetCrypto. |

### KEEP unchanged (DIDComm-specific JOSE composition = the GAP)
`Jose/Base64Url.cs`, `Jose/Jwk.cs`, `Jose/JoseAlgorithms.cs`, `Jose/EnvelopeKind.cs`, `Jose/EnvelopeDetector.cs`, `Jose/Encryption/{ApuComputer,ApvComputer,JweProtectedHeader,RecipientWrap}.cs`, `Jose/Signing/JwsProtectedHeader.cs`, and `Crypto/ICryptoProvider.cs` (the JOSE-string-dispatched facade) + `Crypto/Aead/IAead.cs` (collapse to a thin enc→cipher+sizes shim once the 3 concrete AEADs are gone).

### ZERO edits (firewalled by the facade)
`Composition/{EnvelopeReader,EnvelopeWriter,ForwardWrapper}.cs`, `Protocols/Routing/ForwardProcessor.cs`, `Protocols/Rotation/{FromPriorBuilder,FromPriorValidator}.cs`, `Facade/DidCommClient.cs`, DI extensions.

## 4. Implementation plan (ordered; do NOT start until approved)

**Phase A — Pre-flight verification (no edits)**
- [ ] A1. Confirm `NetDid.Core 2.0.0` + `NetCrypto 1.0.0` restore cleanly together (no diamond conflicts on NetCid/Microsoft.IdentityModel.Tokens). NetDid 2.0 → NetCid 1.6.0; current props pin nothing conflicting.
- [ ] A2. Spot-confirm the ConcatKdf + sign-format contracts on the shipped `NetCrypto.dll` (already strongly evidenced; this is the byte-equivalence gate that only the vectors fully close — see Phase D).

**Phase B — Package re-targeting**
- [ ] B1. `Directory.Packages.props`: bump `NetDid.* 1.3.0 → 2.0.0`; add `NetCrypto 1.0.0`; **remove** `NSec.Cryptography`. Update the explanatory comments.
- [ ] B2. `DidComm.Core.csproj`: add `NetCrypto` reference; remove the direct `NSec` reference; update comments.

**Phase C — Code migration** (Crypto layer → Jose shims → resolver/adapter; in this order)
- [ ] C1. `DefaultCryptoProvider.cs`: re-point the wrapped provider to NetCrypto; route AEAD/KeyWrap to NetCrypto ciphers/`AesKeyWrap`; rename the internal seam. **Preserve public ctor/method surface.**
- [ ] C2. Delete the 4 hand-rolled crypto files (§2); collapse `IAead` to the enc-dispatch+sizes shim.
- [ ] C3. Re-point `EcdhEsKdf`, `Ecdh1PuKdf`, `EphemeralKeyPair`, `KeyTypeMapper` using-aliases to NetCrypto.
- [ ] C4. `JwkConversion.cs`: swap to `NetCrypto.JwkConverter`; **keep OKP-32 guard**; **add explicit `EcPointValidator.EnsureOnCurve`** (G2).
- [ ] C5. `JweBuilder/JweParser/JwsBuilder/JwsParser`: route extraction through didcomm `JwkConversion`; re-point the crypto-provider seam.
- [ ] C6. `NetDidKeyService.cs`: crypto calls → NetCrypto (`FromMulticodec`, `ToPublicJwk`); keep resolution on NetDid + NetCid.
- [ ] C7. `NetDidKeyStoreSecretsResolver.cs`: keystore/JWK types → NetCrypto.
- [ ] C8. Re-point NetDid-crypto imports in tests + samples (`samples/_shared/PeerIdentityFactory.cs`, `samples/02-Cookbook/*`, ~12 test files). Split DID-creation (stays NetDid.Method.*) from crypto (→ NetCrypto).

**Phase D — Verification (the equivalence gate)**
- [ ] D1. Full `DidComm.Core.Tests` + `DidComm.InteropTests` green — especially KDF known-answer (`EcdhEsKdfTests`, `Ecdh1PuKdfTests`), AES-KW RFC 3394 vectors, A256CBC-HS512/A256GCM/XC20P round-trips, JWS sign/verify per curve incl. P-521/ES512.
- [ ] D2. Cross-impl interop fixtures (SICPA/askar 1PU + anoncrypt vectors) — the only thing that fully proves ConcatKdf + 1PU `Ze‖Zs`/tag-framing byte-equivalence.
- [ ] D3. Negative-path: off-curve/forged epk rejected (FR-ENC-03), self-consistent-but-forged JWK whose x/y disagree rejected, tamper-the-tag → `CryptographicException`, DER-vs-P1363 wrong-format → verify=false (per CLAUDE.md identity/integrity guidance).
- [ ] D4. Build confirms the firewalled consumers (§2 "zero edits") compile untouched.

**Phase E — Docs & housekeeping**
- [ ] E1. Update **PRD §3.1/§3.2** to state NetCrypto is the crypto substrate; NetDid.Core is resolution-only; NSec direct ref removed. (PRD is the target — it currently mandates NetDid 1.3.0 as the substrate.)
- [ ] E2. Update `codebase-architecture.md` if present, `CHANGELOG.md`, and `tasks/lessons.md` (capture the version-split + the "DataProofs is off-path for DIDComm" finding).
- [ ] E3. File foundation-side issues for G1–G3 (JOSE enveloping in dataproofs-dotnet; document EcPointValidator auto-check; HSM key-agreement) — these are **dependency-repo** items, not didcomm bugs.

## 5. Risks & how they're handled
- **ConcatKdf / 1PU byte-equivalence (highest):** contract verified in XML; fully closed only by D1+D2 interop vectors. If a vector fails → diff KEK bytes against the NetDid-backed path before cutover.
- **EcPointValidator (G2):** removed as a risk by calling it explicitly in didcomm (not relying on the foundation).
- **Facade surface drift:** keep `DefaultCryptoProvider` public name/ctor stable → firewalled consumers untouched (D4).
- **Branch hygiene:** all work on a `feat/` branch off `main` (never commit to main).

## 6. Out of scope
- Adding `DataProofsDotnet` to didcomm (it's off-path — Gap G1 is a *foundation* item).
- Eliminating libsodium/NSec transitively (only the direct ref is removed — G6).
- Building a foundation JOSE library (G1 — belongs in dataproofs-dotnet).
