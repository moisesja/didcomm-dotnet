**Design decisions worth flagging**
1. **Breaking facade return type.** Phase 3's `PackEncryptedAsync` returned
   `Task<string>`; Phase 4 needed to surface the transport URI. Adopted
   structured `PackEncryptedResult` always (option (a) from the plan) rather
   than overloading or splitting. Phase 3 isn't released so the in-repo
   call-site update was a 6-line search-and-replace.
2. **NetDidServiceEndpointResolver placement.** Moved into `DidComm.Core`
   sibling-to-`NetDidKeyService` rather than the `DidComm.Adapters.NetDid`
   project. `NetDidKeyService` is already in Core, the DI extension already
   references Core, and the IKeyStore→ISecretsResolver bridge is the only
   thing the adapter project ships. Keeping the boundary tight.
3. **Spec re-read on prepend behavior.** Initially appended the mediator's
   own routingKeys after its keyAgreement; backed out after Checkpoint D
   test surfaced the third hop. L-009 captures the rule.
4. **Cookbook section O uses an inline `IServiceEndpointResolver`.** did:peer
   in net-did's current builder doesn't carry routing services, so the
   section mints a mediator inline and patches Bob's routing via a 20-line
   adapter rather than dragging fixture files into the sample.
5. **Mediator secrets reuse Bob's private bytes.** didcomm-python's vendored
   mediator DID Documents do the same ("FIXME — currently a copy-paste from
   Bob's"). Documented in `fixtures/secrets/README.md`.

**Open items / known limitations**
- **Single-recipient forward only.** Multi-recipient `Forward = true` throws
  `InvalidOperationException`. Multi-recipient forward would require either
  per-recipient pack calls (caller's responsibility) or a router that emits
  one wrapped message per recipient (out of scope for Phase 4).
- **Multiple mediator keyAgreement keys.** The expander prepends only the
  first one; the spec text reads "keys" plural but a literal multi-key wrap
  produces a deep onion for no obvious benefit. If Phase 6 interop fixtures
  push back, revisit.
- **`ForwardProcessor` exposes the sync-over-async unpack seam.** Same
  support-boundary caveat as `DidCommClient.UnpackAsync` (PRD §7) —
  documented inline; safe on ASP.NET Core / generic host / console.
