## Batch 1 ‚Äî HIGH: WebSocket SSRF (connect-time IP pinning)  [BEHAVIORAL]

- [ ] **#1 / #2(WS) WebSocket transport SSRF** ‚Äî `src/DidComm.Transports.WebSocket/WebSocketDidCommTransport.cs`
  - Root cause: `Validate()` resolves-then-discards; `ClientWebSocket.ConnectAsync(uri)` re-resolves
    independently ‚Üí DNS-rebinding TOCTOU; fully bypassed when `ResolveDnsNames=false`.
  - Fix: mirror the HTTP transport's pinning. Build a reusable `HttpMessageInvoker` wrapping a
    `SocketsHttpHandler { ConnectCallback = (ctx,ct) => new NetworkStream(await _guard.ConnectAsync(ctx.DnsEndPoint, ct), ownsSocket:true) }`,
    and connect via the `ClientWebSocket.ConnectAsync(uri, invoker, ct)` overload (.NET 7+). The
    handler still does TLS against the original host (SNI/cert unaffected); only the TCP IP is pinned
    to a guard-vetted address. `ConnectAsync` always resolves, so this also closes the
    `ResolveDnsNames=false` WS bypass.
  - Keep the `_options.Connect` custom seam (tests/TestServer own their vetting) and `WebSocketFactory`.
  - Dispose the invoker in `DisposeAsync`.
  - Tests: add a WS test proving a host that resolves to a private IP is refused at connect time even
    with `ResolveDnsNames=false`; add a rebinding-style test (first resolve public, second private) via
    an injected resolver seam.
  - Verify the HTTP `ConnectCallback` pattern in `HttpDidCommBuilderExtensions.cs:48-54` and mirror it.

**Gate 1:** build + test, commit `fix(transports): pin WebSocket connections to vetted IPs (SSRF, FR-TRN)`.

## Batch 3 ‚Äî MEDIUM: identity-trust hardening  [BEHAVIORAL ‚Äî touches PRD + tests]

- [ ] **#4 `from_prior` on unauthenticated envelopes + freshness** ‚Äî `src/DidComm.Core/Facade/DidCommClient.cs:381-391`,
      `src/DidComm.Core/Protocols/Rotation/FromPriorValidator.cs`, `FromPriorClaims.cs`
  - Gate: require `internalResult.Authenticated` (authcrypt skid or signed), not merely `Encrypted`,
    before accepting `from_prior`. Reject (or drop, matching FR-ROT-03 style) otherwise.
  - Freshness: read `exp`/`nbf` into `FromPriorClaims`; if present, validate against `_options.Now()`
    with `ExpiresClockSkew`; reject expired/not-yet-valid; reject implausible far-future `iat`.
  - **PRD:** update FR-ROT-03/05 wording; reconcile FR-ROT-05 "closed" status (document app-vs-library
    responsibility honestly). **Tests:** existing anoncrypt rotation test must move to authcrypt; add
    reject-on-anoncrypt and expired-JWT tests.
- [ ] **#10 Unauthenticated `from` consumed by reply handlers** ‚Äî handlers + `UnpackResult` docs
  - Send paths are already safe (HTTP logs only; WS same-socket re-packs authcrypt to the authenticated
    peer), so this is defense-in-depth: (a) add XML-doc warnings on `UnpackResult.Message`/`From` that
    the value is only trustworthy when `Authenticated`/`NonRepudiation`; (b) **do not** break legitimate
    anoncrypt request/response (e.g. TrustPing `response_requested`) ‚Äî instead ensure no handler newly
    introduces an out-of-band send keyed on unauthenticated `from`. Decision point for review: do we
    want handlers to refuse addressed replies when `!Authenticated`, or keep spec-legal anoncrypt
    responses? Default proposal: keep behavior, add docs + the #4 gate.

**Gate 3:** build + test, commit `fix(rotation): require sender-authentication + freshness for from_prior (FR-ROT-03/05)`.
