# Phase 5 — Transports (FR-TRN-01..12, FR-API-06 413 path)

PRD §9 (lines 392–422) + §12 Phase 5 (lines 594–602) + §14.2 cookbook sections **P/Q/R** (lines 888–903).

## Context

Phase 4 closed routing: `DidCommClient.PackEncryptedAsync(... Forward: true)` now resolves the recipient's `DIDCommMessaging` service, wraps `forward` layers, and surfaces a `ServiceEndpoint` on `PackEncryptedResult`. **Nothing in the repo turns that into bytes-on-the-wire today** — exhaustive grep confirmed there is no `IDidCommTransport`, no `SendAsync`, no `MapDidCommEndpoint`, and no `TransportException` (the latter being a gap in the FR-API-07 exception taxonomy).

Phase 5 adds the HTTPS and WebSocket transports and the ASP.NET Core receive endpoint, all gated on `MaxReceiveBytes → 413` (FR-API-06 transport path) and the spec's reverse rule for HTTP redirects (307 follow, 308/301 surface). Cookbook sections P/Q/R land in the same phase per the §12 ratchet.

Existing patterns to reuse: builder-chain DI (`DidCommBuilder.UseNetDidResolver` / `UseSecretsResolver`), per-section cookbook structure (Section_O is the template), `Microsoft.AspNetCore.TestHost` already pinned in `Directory.Packages.props` for this exact use, exception hierarchy under `DidComm.Exceptions` with the 3-ctor pattern.

## Architecture

Four projects, three new:

```
src/
  DidComm.Core/                          (existing — adds Transports/* abstractions + SendAsync)
  DidComm.Transports.Http/               NEW — client HTTPS sender
  DidComm.Transports.WebSocket/          NEW — client WS sender
  DidComm.AspNetCore/                    NEW — MapDidCommEndpoint + MapDidCommWebSocket
  DidComm.Extensions.DependencyInjection/(existing — unchanged here; each transport project adds its own .UseXxxTransport() ext)
tests/
  DidComm.Core.Tests/Transports/         router + abstractions + SendAsync wiring
  DidComm.InteropTests/Transports/       HttpMessageHandler stubs + TestServer round-trips + in-memory WebSocket pair
samples/02-Cookbook/Sections/
  Section_P_SendOverTransport.cs
  Section_Q_ReceiveHttp.cs
  Section_R_ReceiveWebSocket.cs
```

**Boundary rule:** transport client (sender) packages MUST NOT depend on `Microsoft.AspNetCore.*`. A console agent that only sends should not drag the ASP.NET host in. The `DidComm.AspNetCore` project owns every ASP.NET Core dep.

**Resilience:** **Polly** (`Microsoft.Extensions.Http.Polly` + `Polly`) for HTTP retry + timeout + circuit-breaker (FR-TRN-08). Already battle-tested; standard `IHttpClientFactory` integration via `AddPolicyHandler`. WebSocket reconnect uses `Polly`'s `RetryPolicy` for the 1s/30s/0.5-jitter exponential backoff (DD-05 / FR-TRN-11). Adds `Polly` to `Directory.Packages.props`.

**Receiver seam:** **No abstraction in Phase 5.** `MapDidCommEndpoint` and `MapDidCommWebSocket` accept the handler inline at registration time:

```csharp
app.MapDidCommEndpoint("/didcomm", async (UnpackResult result, CancellationToken ct) =>
{
    // host's logic here — typically dispatch by result.Message.Type
});
```

Phase 6 will ship the proper `IProtocolHandler` registry (FR-PROTO-03) and add a parameterless overload (`app.MapDidCommEndpoint("/didcomm")`) that resolves the registry from DI. The Phase 5 inline-delegate overload remains valid forever — Phase 6's registry-aware overload is additive. We avoid introducing `IDidCommMessageReceiver` now because (a) Phase 5 has no protocol handlers to dispatch to yet and (b) prematurely freezing the receiver shape risks over-fitting before Phase 6 (lesson L-009).

## Checkpoint plan

Six checkpoints, each gated on green tests + `dotnet build /warnaserror` clean before the next one starts. Matches Phase 4 cadence.

### A — Core transport abstractions (FR-TRN-01..03, FR-API-07)

**Adds in `src/DidComm.Core/`:**
- `Exceptions/TransportException.cs` — derives from `DidCommException`, 3-ctor pattern (file-shape match: existing `DidResolutionException.cs`).
- `Transports/IDidCommTransport.cs` — `string Scheme`, `bool CanHandle(string uri)`, `Task<TransportResult> SendAsync(TransportRequest req, CancellationToken ct)`.
- `Transports/TransportRequest.cs` (record: `Uri Endpoint`, `ReadOnlyMemory<byte> Payload`, `string MediaType`).
- `Transports/TransportResult.cs` (record: `bool Accepted`, `int? HttpStatusCode`).
- `Transports/ITransportRouter.cs` + `TransportRouter.cs` (internal default; picks by `Uri.Scheme` case-insensitive; throws `TransportException` when no transport handles a scheme).
- `Transports/SendOptions.cs` — mirrors `PackEncryptedOptions` minus `Forward` (always `true` for `SendAsync`), adds `ServiceEndpointOverride` (optional, for senders that already know the URI without DID resolution).
- `Transports/SendResult.cs` (record: `PackEncryptedResult Packed`, `TransportResult Transport`, `string TransportEndpointUsed`).
- `Facade/DidCommClient.cs` — add `Task<SendResult> SendAsync(Message, SendOptions, CancellationToken)`. Internally: `PackEncryptedAsync(... Forward: true)`, pick endpoint from `PackEncryptedResult.ServiceEndpoint`, dispatch via injected `ITransportRouter`. New 5-arg public ctor + ctor overload accepting the router.

**Tests** (`tests/DidComm.Core.Tests/Transports/`):
- `TransportRouterTests` — dispatch by scheme; unknown scheme throws `TransportException`; case-insensitive scheme match; null endpoint refused.
- `DidCommClientSendAsyncTests` — happy path with a stub `IDidCommTransport`; refuses when no router is registered; surfaces `PackEncryptedResult` via `SendResult.Packed`.

### B — HTTPS sender (FR-TRN-04..06, FR-TRN-08)

**New project: `src/DidComm.Transports.Http/`**
- Targets `net10.0`, references `DidComm.Core` + `DidComm.Extensions.DependencyInjection` + `Microsoft.Extensions.Http` + `Microsoft.Extensions.Http.Polly` (which transitively pulls `Polly`).
- `HttpDidCommTransport` — uses `IHttpClientFactory` (named client `"didcomm"`); sets `Content-Type` from `TransportRequest.MediaType`; treats any 2xx as `Accepted=true` (FR-TRN-05); follows **307 only** by configuring `HttpClientHandler.AllowAutoRedirect = false` and re-issuing manually for 307 with the same method+body; surfaces 301/308 + non-2xx as `TransportException` with a status-code field.
- `HttpTransportOptions` — `TimeSpan RequestTimeout = 30s`, `int MaxRetryAttempts = 3`, `TimeSpan RetryBaseDelay = 1s`, `int CircuitBreakerFailureThreshold = 5`, `TimeSpan CircuitBreakerOpenDuration = 30s`, `string[] AllowedSchemes = {"https"}` (HTTP off by default; opt-in for tests).
- `HttpDidCommBuilderExtensions.UseHttpTransport(this DidCommBuilder, Action<HttpTransportOptions>?)` — registers the transport in the DI router slot AND wires Polly policies via `AddPolicyHandler(...)`: `WaitAndRetryAsync` (exponential w/ jitter, transient HTTP errors + 5xx + `TaskCanceledException` timeout) chained with `CircuitBreakerAsync` and the `Timeout` policy. Policies are constructed from `HttpTransportOptions`.

**Tests** (`tests/DidComm.InteropTests/Transports/HttpTransportSendTests.cs`):
- `HttpMessageHandler` stub returning 200/202/204 ⇒ `Accepted=true`.
- 307 redirected ⇒ followed, final status surfaced.
- 308 / 301 ⇒ `TransportException`.
- 500 ⇒ retried `MaxRetryAttempts` times then throws.
- Wrong scheme ⇒ `CanHandle` false; router skips it.

### C — ASP.NET Core HTTP receive (FR-TRN-07, FR-API-06 413)

**New project: `src/DidComm.AspNetCore/`**
- Targets `net10.0`, references `DidComm.Core` + the ASP.NET Core shared framework (`<FrameworkReference Include="Microsoft.AspNetCore.App" />` — standard pattern; no transitive NuGets needed).
- `DidCommEndpointRouteBuilderExtensions.MapDidCommEndpoint(IEndpointRouteBuilder, string pattern, Func<UnpackResult, CancellationToken, Task> onReceive)` — minimal-API style; handler passed inline (Phase 6 adds a parameterless overload that resolves an `IProtocolHandler` registry from DI).
  - Validates `Content-Type` matches one of the DIDComm media types; mismatched ⇒ 415.
  - Streams body with a hard cap at `DidCommOptions.MaxReceiveBytes`; over ⇒ 413 (FR-API-06).
  - Unpacks via injected `DidCommClient.UnpackAsync`.
  - Invokes `onReceive(unpackResult, ct)`; returns **202** (FR-TRN-07).
  - On `MalformedMessageException` / `CryptoException` ⇒ 400; on `TransportException` ⇒ 502.

**Tests** (`tests/DidComm.InteropTests/Transports/AspNetCoreReceiveRoundTripTests.cs`):
- Build a `WebApplication` via `WebApplication.CreateBuilder` + `Microsoft.AspNetCore.TestHost.TestServer` (already in `Directory.Packages.props`).
- Round-trip: Alice packs (no Forward — direct send) → `HttpDidCommTransport.SendAsync` against the TestServer → TestServer unpacks → receiver records the message → assert content matches.
- 413: payload > `MaxReceiveBytes` ⇒ TestServer returns 413.
- 415: wrong Content-Type ⇒ 415.

### D — WebSocket sender (FR-TRN-09..11)

**New project: `src/DidComm.Transports.WebSocket/`**
- Targets `net10.0`, references `DidComm.Core` + `DidComm.Extensions.DependencyInjection`. Uses `System.Net.WebSockets.ClientWebSocket` (BCL).
- `WebSocketDidCommTransport`
  - One WebSocket **binary** message per packed DIDComm envelope (FR-TRN-09 — JOSE serializations are UTF-8 bytes; binary keeps frame size accurate and avoids text-validation overhead).
  - Frame reassembly: send-side uses `EndOfMessage=true` on the last segment; receive-side (Checkpoint E) loops until `EndOfMessage=true`.
  - Connection pool keyed by `Uri.Authority + Path`; reconnect with exponential backoff (1s base, 30s cap, 0.5 jitter — DD-05 per PRD §9.3 / FR-TRN-11).
  - `event EventHandler<WebSocketLifecycleEventArgs> Lifecycle` — Connected / Disconnected / SendFailed.
  - One-way: send only; receive path lives in the AspNetCore extension (Checkpoint E).
- `WebSocketTransportOptions` — `TimeSpan ConnectTimeout = 10s`, `TimeSpan SendTimeout = 30s`, `int MaxReconnectAttempts = 5`, deterministic-clock seam (`Func<DateTimeOffset> Clock`) for tests.
- `WebSocketDidCommBuilderExtensions.UseWebSocketTransport(...)`.

**Tests** (`tests/DidComm.InteropTests/Transports/WebSocketTransportSendTests.cs`):
- Use `Microsoft.AspNetCore.TestHost.TestServer.CreateWebSocketClient()` to spin up an in-process WS endpoint and exercise send.
- Frame reassembly: server splits the response across multiple physical frames; client receives one logical envelope. (Send-side is naturally single-frame for typical payloads; the reassembly cover comes in Checkpoint E.)
- Reconnect: drop the connection mid-test; transport reconnects and resumes.

### E — ASP.NET Core WebSocket receive (FR-TRN-09..10)

**Adds to `src/DidComm.AspNetCore/`:**
- `DidCommEndpointRouteBuilderExtensions.MapDidCommWebSocket(IEndpointRouteBuilder, string pattern, Func<UnpackResult, CancellationToken, Task> onReceive)`.
  - Accept WebSocket; loop reading frames until `EndOfMessage=true`; concatenate into one logical envelope; unpack; invoke `onReceive(unpackResult, ct)`.
  - One-way (FR-TRN-10): server does not send protocol replies on the same socket.
  - Honors `MaxReceiveBytes` per logical envelope ⇒ close socket with 1009 (Message Too Big) on overflow.

**Tests** (`tests/DidComm.InteropTests/Transports/AspNetCoreWebSocketRoundTripTests.cs`):
- TestServer round-trip: client sends one packed envelope as a multi-frame binary message; server reassembles, unpacks, receiver records.
- Oversize message ⇒ socket closed with 1009.

## User-confirmed decisions

1. **Project split** — keep three projects (`DidComm.Transports.Http`, `DidComm.Transports.WebSocket`, `DidComm.AspNetCore`). CLI senders pay zero ASP.NET cost.
2. **Resilience** — **Polly** via `Microsoft.Extensions.Http.Polly` for HTTP retry + timeout + circuit-breaker (FR-TRN-08). Adds `Polly` to `Directory.Packages.props`. Standard `AddPolicyHandler` chain on the named `IHttpClientFactory` client. WebSocket reconnect (FR-TRN-11) uses Polly's `RetryPolicy` for the 1s/30s/0.5-jitter exponential backoff (DD-05).
3. **Cookbook hosting** — `Microsoft.AspNetCore.TestHost.TestServer` in-process for sections Q/R (no Kestrel port, offline-safe, matches `Directory.Packages.props` intent).
4. **Receiver seam** — **no `IDidCommMessageReceiver` abstraction in Phase 5.** `MapDidCommEndpoint` and `MapDidCommWebSocket` take an inline `Func<UnpackResult, CancellationToken, Task>` handler at registration. Phase 6 lands the proper `IProtocolHandler` registry (FR-PROTO-03) and adds parameterless overloads that resolve the registry from DI — additive, no breaking change.

**Design decisions worth flagging**
1. **Polly v8 used directly (not `Microsoft.Extensions.Http.Resilience`).** The
   `HttpResiliencePipelineFactory` builds a `ResiliencePipeline<HttpResponseMessage>`
   from a `HttpTransportOptions` instance and the `HttpDidCommTransport` drives it
   over `IHttpClientFactory.CreateClient("didcomm")`. Skipping the Resilience
   extensions package keeps the dep tree tighter and the policy construction
   visible.
2. **307 followed manually inside the transport, not the handler.** Polly handles
   transient retries; the transport itself owns the FR-TRN-06 redirect rule. The
   handler factory disables `AllowAutoRedirect` so 301/308 surface as real
   responses the transport can refuse.
3. **WebSocket factory / connect seams.** `WebSocketTransportOptions.WebSocketFactory`
   + `Connect` let the cookbook + InteropTests substitute a
   `TestServer.CreateWebSocketClient()` socket without touching the production
   `ClientWebSocket.ConnectAsync` path. Real hosts leave both null and get the
   default `ClientWebSocket` behavior.
4. **No `IDidCommMessageReceiver` interface (per user-confirmed decision #4).**
   `MapDidCommEndpoint` / `MapDidCommWebSocket` take an inline
   `Func<UnpackResult, CancellationToken, Task>` handler. Phase 6 will add a
   parameterless overload that resolves the FR-PROTO-03 `IProtocolHandler`
   registry from DI — purely additive.
5. **Cookbook hosts an in-process `TestServer` rather than Kestrel.** Sections
   P/Q/R stay offline-safe and don't pick a port. The Cookbook csproj brings in
   `Microsoft.AspNetCore.TestHost` + the AspNet shared framework reference for
   this reason.
6. **`MaxReceiveBytes` already existed for unpack (Phase 3).** Phase 5 reuses
   the same `DidCommOptions.MaxReceiveBytes` knob from the HTTP and WS receive
   paths, returning 413 / 1009 instead of `MalformedMessageException`. No new
   option needed.

**Open items / known limitations**
- **Single-receiver per endpoint.** `MapDidCommEndpoint` / `MapDidCommWebSocket`
  hand the unpacked message to one inline handler. Phase 6 adds a registry
  overload; both shapes coexist forever.
- **No 408 handling distinct from 5xx.** The Polly retry predicate treats
  `HttpStatusCode.RequestTimeout` (408) as transient — same as 5xx. Good
  enough for Phase 5; if Phase 6 interop pressure wants finer control,
  expose the predicate on `HttpTransportOptions`.
- **WebSocket transport hardcodes binary frames.** FR-TRN-09 ambiguity (binary
  vs text) is decided in favor of binary because JOSE envelopes are
  arbitrary UTF-8 bytes; if a peer demands text frames, expose a `FrameType`
  option.
- **No STOMP-over-WebSocket (FR-TRN-12, MAY).** Deliberately deferred —
  no fixture pressure for it in Phase 5.
