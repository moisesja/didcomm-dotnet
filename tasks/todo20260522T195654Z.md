`DidComm.Core` internal layout:

```
src/DidComm.Core/
  DidComm.Core.csproj
  Crypto/
    ICryptoProvider.cs              # contract (PRD ¬ß3.3)
    DefaultCryptoProvider.cs        # composition root, delegates to primitives
    Primitives/
      KeyAgreement/                 # X25519, P-256, P-384, P-521
        X25519KeyAgreement.cs       # NSec
        NistEcdhKeyAgreement.cs     # System.Security.Cryptography
      Signing/
        Ed25519Signer.cs            # NSec
        EcdsaSigner.cs              # ES256 via S.S.C; ES256K via BouncyCastle
      Aead/
        AesCbcHmacSha512.cs         # System.Security.Cryptography (manual compose)
        AesGcmAead.cs               # System.Security.Cryptography
        XChaCha20Poly1305Aead.cs    # NSec
      KeyWrap/
        AesKeyWrap.cs               # System.Security.Cryptography
      Kdf/
        ConcatKdf.cs                # RFC 7518 ¬ß4.6 + 1PU
      PointValidation/
        NistPointValidator.cs       # FR-ENC-03 stub (used in Phase 2)
  Encoding/
    Base64Url.cs                    # no-pad codec
  Jose/
    Jwk.cs                          # data model
    JwkConverter.cs                 # JWK ‚Üî key conversions
  GlobalUsings.cs                   # implicit usings
```

Notes on choices:

- **NSec** owns X25519, Ed25519, XC20P (per PRD ¬ß3.1).
- **System.Security.Cryptography** owns P-256/384/521 ECDH+ECDSA, AES-CBC, AES-GCM, AES-KW, HMAC-SHA512, SHA-256.
- **BouncyCastle** is used **only** for secp256k1 (ES256K). All other curves use the BCL.
- `AesCbcHmacSha512` is composed manually from `Aes` + `HMACSHA512` (per RFC 7518 ¬ß5.2.5); the AEAD interface returns `(ciphertext, tag, iv)`.
- `ConcatKdf` is a single class with two entry points: `DeriveEcdhEs(z, alg, apu, apv, keyLen)` and `DeriveEcdh1Pu(ze, zs, alg, apu, apv, tag, keyLen)`. Both use the same underlying `Concat(Z, otherInfo)` machinery and differ only in `Z` composition and the inclusion of the AEAD tag in `SuppPubInfo` (FR-ENC-15 hook ‚Äî but the *ordering* of encrypt-then-derive lives in the envelope code in Phase 2, not in the KDF).

### 2.3 Primitives

- [ ] `Primitives/KeyAgreement/X25519KeyAgreement.cs`:

- [ ] `Primitives/KeyAgreement/NistEcdhKeyAgreement.cs`:

- [ ] `Primitives/Aead/AesCbcHmacSha512.cs`:
  - Implement RFC 7518 ¬ß5.2.5 (split key, encrypt-then-MAC over AAD ‚Äñ IV ‚Äñ ciphertext ‚Äñ AL).

- [ ] `Primitives/Aead/AesGcmAead.cs`: standard BCL.
- [ ] `Primitives/Aead/XChaCha20Poly1305Aead.cs`: NSec `AeadAlgorithm.XChaCha20Poly1305`.
- [ ] `Primitives/KeyWrap/AesKeyWrap.cs`: RFC 3394 via BCL `AesKeyWrap` (System.Security.Cryptography is available in .NET 8+; if the API is missing for .NET 10 preview I fall back to BCL's `KeyWrap` implementation that ships with BCL ‚Äî verify on first build).
- [ ] `Primitives/Kdf/ConcatKdf.cs`:
  - Single `Concat(Z, algorithmId, apu, apv, suppPubInfo, keyDataLen)` per RFC 7518 ¬ß4.6 / NIST SP 800-56A ¬ß5.8.1.

- [ ] `Primitives/PointValidation/NistPointValidator.cs`:
  - For each NIST curve, verify `(x, y)` lies on the curve (substitute into `y¬≤ = x¬≥ + ax + b mod p`) and is not the identity.
  - Used in Phase 2 unpack; here we ship the function and a unit test that crafts an off-curve point and asserts rejection (FR-ENC-03 acceptance criterion satisfied in advance).

## 5. Risk register

- **R1. `Portable.BouncyCastle 1.9.0` on .NET 10.** The package targets `netstandard2.0` and is known to work, but a `TreatWarningsAsErrors=true` build can trip on obsoletion warnings inside BCL types we reference. Mitigation: if warnings surface, isolate BCL usage in a single `EcdsaSigner` file and add `#pragma warning disable` with a comment naming the BCL warning IDs ‚Äî never a blanket suppression in `.editorconfig`.
- **R2. `AeadAlgorithm.XChaCha20Poly1305` availability in NSec 24.4.0.** Confirmed in the 24.x API; if it later moves namespaces, isolate behind `XChaCha20Poly1305Aead`.
- **R3. NSec on Windows CI.** NSec ships native libsodium binaries via runtime IDs; CI must restore on both Linux and Windows. If Windows restore fails, add `RuntimeIdentifiers` to the test project.
- **R4. RFC 7518 ¬ßC example fidelity.** The KAT in the RFC is for `ConcatKDF(SHA-256, Z, OtherInfo)` with specific inputs; I will transcribe the example values byte-for-byte from the RFC and cite the source in the test file.

## 6. Status

**STATUS: PAUSED ‚Äî blocked on net-did 1.3.0.**

After scaffolding began, we identified that ~30% of Phase 0's planned crypto primitives belong in net-did as generic SSI infrastructure (raw ECDH, P-521, ECDSA P1363, off-curve validation, Concat KDF), not in didcomm-dotnet. Filing them in didcomm-dotnet would have meant code duplication across every future .NET SSI library (didcomm-dotnet, zcap-dotnet, VC libs).

Five critical issues filed against `moisesja/net-did`:

- moisesja/net-did#60 ‚Äî Raw ECDH key agreement (X25519/P-256/P-384/P-521) returning unprocessed `Z`.
- moisesja/net-did#61 ‚Äî P-521 as a first-class `KeyType`.
- moisesja/net-did#62 ‚Äî ECDSA IEEE P1363 signature format option.
- moisesja/net-did#63 ‚Äî Off-curve EC public key rejection in `JwkConverter` (security defect).
- moisesja/net-did#64 ‚Äî Concat KDF (RFC 7518 ¬ß4.6 / NIST SP 800-56A ¬ß5.8.1) public helper.

### What's been done in this repo

- PRD ¬ß3.1 updated: `Portable.BouncyCastle` ‚Üí `NBitcoin.Secp256k1` for consistency with net-did.
- `Directory.Packages.props` updated to match.
- `DidComm.sln` + project scaffolding (`src/DidComm.Core`, `tests/DidComm.Core.Tests`, `tests/DidComm.InteropTests`) created. **No source files yet** ‚Äî the crypto layer was intentionally not started because it consumes the net-did extensions above.

### Resumption plan (when net-did 1.3.0 ships)

1. Bump `Directory.Packages.props` `NetDid.Core` version to whichever release includes #60‚Äì#64.
2. Write the **didcomm-dotnet-specific** crypto layer on top of the net-did substrate:
   - `Encoding/Base64Url.cs` ‚Äî wraps `NetCid.Multibase` (already net-did's transitive dep).
   - `Jose/Jwk.cs` ‚Äî light DIDComm-shaped wrapper over `Microsoft.IdentityModel.Tokens.JsonWebKey` and `NetDid.Core.Jwk.JwkConverter`.
   - `Crypto/Aead/AesCbcHmacSha512.cs` ‚Äî RFC 7518 ¬ß5.2.5 (JOSE-specific composition, stays here).
   - `Crypto/Aead/AesGcmAead.cs` ‚Äî BCL passthrough.
   - `Crypto/Aead/XChaCha20Poly1305Aead.cs` ‚Äî NSec passthrough.
   - `Crypto/KeyWrap/AesKeyWrap.cs` ‚Äî RFC 3394; BCL passthrough.
   - `Crypto/Kdf/Ecdh1PuKdf.cs` ‚Äî thin wrapper that composes `Ze ‚Äñ Zs` and routes through net-did's `ConcatKdf` with the AEAD tag in `SuppPubInfo`.
   - `Crypto/ICryptoProvider.cs` / `Crypto/DefaultCryptoProvider.cs` ‚Äî DIDComm's JOSE-shaped surface, delegating to net-did for sign/verify/key-agreement and to local code for AEADs/KW/1PU.
3. Tests, CI, and the InteropTests skeleton land at the same time.

### Why this pause is the right call

- **Don't duplicate.** Phase 0 would have shipped ~150 LOC of crypto primitives that net-did would later need to ship anyway. Two implementations = two attack surfaces.
- **Net-did's off-curve gap is a security defect.** Independent of DIDComm, any net-did consumer that resolves a DID Document and uses its EC verification methods today is exposed to invalid-curve attacks (see #63). Fixing it once in net-did fixes it everywhere.
- **The PRD's NFR-08 (CI green) doesn't gate on having a binary right now** ‚Äî it gates on the eventual phase exit. Pausing is cheap.
