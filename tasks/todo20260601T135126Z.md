### A — Tolerate a trailing URL fragment in query parsing  (src/DidComm.Core/Protocols/OutOfBand/OutOfBand.cs)
`TryGetQueryParameter` splits the substring after `?` on `&` but never strips a `#fragment`.
For `https://x/y?_oob=<b64>#frag` the `#frag` is folded into the value → base64url decode
throws `FormatException`. Fix: cut the query at the first `#` before splitting. One change
covers both `_oob` and `_oobid` (short-form) paths.

```csharp
var query = url[(queryStart + 1)..];
// A fragment (#...) is not part of the query; drop it so an _oob/_oobid value followed by a
// fragment still decodes instead of failing base64url on the trailing "#frag".
var hash = query.IndexOf('#');
if (hash >= 0)
    query = query[..hash];
```
+ unit test `FromUrl_ignores_a_trailing_url_fragment`.

### B — Lock in web_redirect on-wire shape through serialization  (tests/DidComm.Core.Tests/Protocols/OutOfBand/OutOfBandTests.cs)
Existing tests exercise `AddWebRedirect`→`ReadWebRedirect` only in-memory. `web_redirect`
rides in `[JsonExtensionData]` and is written with default (not `DidCommJson`) options, so a
pack→reparse→read test guards the `{"status","redirectUrl"}` shape (FR-OOB-05) against future
serializer changes. Test only — no source change.

```csharp
[Fact]
public void AddWebRedirect_survives_a_plaintext_serialize_round_trip()
{
    var message = Message.Empty().WithFrom("did:example:verifier").WithTo("did:example:prover").Build();
    OutOfBandApi.AddWebRedirect(message, new WebRedirect("OK", "https://example.com/done"));

    var json = DidComm.Composition.EnvelopeWriter.PackPlaintext(message);            // real plaintext path
    var reparsed = JsonSerializer.Deserialize<Message>(json, DidComm.Json.DidCommJson.Default)!;

    var read = OutOfBandApi.ReadWebRedirect(reparsed);
    read.Should().NotBeNull();
    read!.Status.Should().Be("OK");
    read.RedirectUrl.Should().Be("https://example.com/done");
}
```
(`DidCommJson` is internal but `InternalsVisibleTo DidComm.Core.Tests` is set; `PackPlaintext`
is likewise reachable — both already used by sibling tests.)

### C — Correct the IOptions<TraceOptions> comment  (src/DidComm.Extensions.DependencyInjection/DidCommBuilder.cs)
On closer analysis the registration is **not** fragile: MS DI prefers an exact closed
`IOptions<TraceOptions>` descriptor over the open-generic `IOptions<>` fallback regardless of
registration order, so this is order-independent and correct. The only defect is the comment,
which implies the mechanism matches `AddOptions<ProblemReportOptions>()` (it doesn't — this is a
manual `Options.Create` wrapper). **Comment-only change; the two registration lines stay as-is.**

```csharp
Services.TryAddSingleton(options);
// Also expose the validated instance as IOptions<TraceOptions> for consumers that prefer the
// Options pattern (ProblemReportOptions offers the same via AddOptions<T>()). Options.Create
// wraps the SAME instance registered above, so both resolve to identical values; the container
// prefers this closed IOptions<TraceOptions> over the open-generic IOptions<> fallback, so it is
// order-independent. TryAdd keeps the "first call wins" idempotency this method documents.
Services.TryAddSingleton<Microsoft.Extensions.Options.IOptions<TraceOptions>>(
    Microsoft.Extensions.Options.Options.Create(options));
```
