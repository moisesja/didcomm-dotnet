## 1. What changed since the original plan

The original Phase 0 plan had `DidComm.Core` owning **all** the crypto primitives (key agreement per curve, signing per curve, AEADs, key wrap, KDF, point validation, JWK conversion). With `NetDid 1.3.0`, **about half of that surface now lives upstream as generic SSI substrate** and `DidComm.Core` only owns the **JOSE-specific composition layer**.

| Original Phase 0 piece | Outcome |
|---|---|
| `Crypto/Primitives/KeyAgreement/X25519KeyAgreement.cs` | **Deleted from plan** — `NetDid.ICryptoProvider.DeriveSharedSecret(KeyType.X25519, …)`. |
| `Crypto/Primitives/KeyAgreement/NistEcdhKeyAgreement.cs` | **Deleted** — same call for `P256`/`P384`/`P521`. |
| `Crypto/Primitives/Signing/Ed25519Signer.cs` | **Deleted** — `NetDid.ICryptoProvider.Sign(KeyType.Ed25519, …)`. |
| `Crypto/Primitives/Signing/EcdsaSigner.cs` (incl. ES256K via BouncyCastle) | **Deleted** — `NetDid.ICryptoProvider.Sign(KeyType.{P256,P384,P521,Secp256k1}, …, EcdsaSignatureFormat.IeeeP1363)`. |
| `Crypto/Primitives/Kdf/ConcatKdf.cs` | **Deleted** — `NetDid.Core.Crypto.Kdf.ConcatKdf.DeriveKey(…)`. |
| `Crypto/Primitives/PointValidation/NistPointValidator.cs` | **Deleted** — `NetDid.Core.Crypto.EcPointValidator.EnsureOnCurve(…)` runs automatically inside `JwkConverter.ExtractPublicKey`. |
| `Jose/JwkConverter.cs` (full reimplementation) | **Reduced** — DIDComm-specific overlay only. `NetDid.Core.Jwk.JwkConverter` (already in net-did) owns the heavy lifting. |
| `Crypto/Primitives/Aead/AesCbcHmacSha512.cs` | **Kept** — RFC 7518 §5.2.5 is a JOSE-only composition. |
| `Crypto/Primitives/Aead/AesGcmAead.cs` | **Kept** — thin BCL wrapper to fit `ICryptoProvider`. |
| `Crypto/Primitives/Aead/XChaCha20Poly1305Aead.cs` | **Kept** — thin NSec wrapper. |
| `Crypto/Primitives/KeyWrap/AesKeyWrap.cs` | **Kept** — RFC 3394; BCL does not expose AES-KW publicly. |
| `Crypto/Primitives/Kdf/Ecdh1PuKdf.cs` (new) | **Added** — wrapper that composes `Z = Ze ‖ Zs` from two `DeriveSharedSecret` calls and threads the AEAD tag into `SuppPubInfo`, then calls net-did's `ConcatKdf`. Strictly JOSE-draft-specific. |
| `Encoding/Base64Url.cs` | **Kept**, but trivial — defer to `Convert.ToBase64String` + url-safe substitution, or wrap `System.Buffers.Text.Base64Url` (available .NET 9+). |

**Net effect:** the Phase 0 crypto layer shrinks from ~12 primitive files to ~5 JOSE-composition files plus an `ICryptoProvider` that delegates 60% of its body to net-did. Direct dependency on `NBitcoin.Secp256k1` is **dropped** — secp256k1 reaches `DidComm.Core` transitively through net-did.

## 3. Step-by-step checklist

### 3.1 Encoding & JOSE foundations

- [ ] `GlobalUsings.cs` — implicit usings for `System.Buffers.Binary`, `System.Security.Cryptography`, `System.Text`.
- [ ] `Encoding/Base64Url.cs` — `Encode(ReadOnlySpan<byte>) → string`, `Decode(string) → byte[]`. No padding, url-safe alphabet. Single allocation per call.
- [ ] `Jose/JoseAlgorithms.cs` — string constants only. No logic.

### 3.2 JWK overlay

- [ ] `Jose/Jwk.cs` — A DIDComm-shaped JWK record. **Decision needed**: do we model it as a plain `record` with `Kty`/`Crv`/`X`/`Y`/`D`/`Kid`/`Alg`/`Use`, or do we adopt `Microsoft.IdentityModel.Tokens.JsonWebKey` (net-did's choice) directly? The latter avoids a parallel hierarchy; the former lets us preserve unknown headers verbatim (FR-MSG-15). **Recommendation:** plain record now, with a `Dictionary<string, JsonElement>` `AdditionalData` bag. Re-evaluate in Phase 1 if FR-MSG-15 forces a richer model.
- [ ] `Jose/JwkConversion.cs` — shims: `ToNetDidJwk(Jwk) → JsonWebKey`, `FromNetDidJwk(JsonWebKey) → Jwk`, plus `ExtractPublicKey(Jwk) → (KeyType, byte[])` which delegates to net-did. **Off-curve validation happens automatically** in `JwkConverter.ExtractPublicKey`.

### 3.3 AEAD primitives (JOSE-composition layer)

- [ ] `Crypto/Aead/IAead.cs`:
  ```csharp
  internal interface IAead
  {
      (byte[] Ciphertext, byte[] Tag) Encrypt(ReadOnlySpan<byte> key, ReadOnlySpan<byte> iv, ReadOnlySpan<byte> aad, ReadOnlySpan<byte> plaintext);
      byte[] Decrypt(ReadOnlySpan<byte> key, ReadOnlySpan<byte> iv, ReadOnlySpan<byte> aad, ReadOnlySpan<byte> ciphertext, ReadOnlySpan<byte> tag);
      int KeySizeBytes { get; }
      int IvSizeBytes { get; }
      int TagSizeBytes { get; }
  }
  ```
- [ ] `Crypto/Aead/AesCbcHmacSha512.cs` — RFC 7518 §5.2.5. 64-byte key (split into 32 MAC + 32 ENC). 16-byte IV. AAD ‖ IV ‖ CT ‖ AL hashed with HMAC-SHA512; first 32 bytes of MAC = `tag`. `CryptographicOperations.FixedTimeEquals` on decrypt (NFR-09).
- [ ] `Crypto/Aead/AesGcmAead.cs` — wraps `System.Security.Cryptography.AesGcm`. 32-byte key, 12-byte IV, 16-byte tag (A256GCM).
- [ ] `Crypto/Aead/XChaCha20Poly1305Aead.cs` — wraps `NSec.Cryptography.AeadAlgorithm.XChaCha20Poly1305`. 32-byte key, 24-byte IV (XC20P), 16-byte tag.

### 3.4 Key wrap

- [ ] `Crypto/KeyWrap/AesKeyWrap.cs` — RFC 3394, 256-bit KEK. ~30 LOC: AES-ECB, six rounds of (`A`, `t = ((n*j)+i)`, AES-encrypt `A ‖ R[i]`, split, XOR), inverse for unwrap. Const-time integrity check on `A` against the IV `A6A6A6A6A6A6A6A6` after unwrap.

### 3.5 ECDH-1PU KDF wrapper

- [ ] `Crypto/Kdf/Ecdh1PuKdf.cs`:
  ```csharp
  internal static class Ecdh1PuKdf
  {
      public static byte[] DeriveKey(
          ICryptoProvider netDidProvider, KeyType curve,
          ReadOnlySpan<byte> senderPrivate, ReadOnlySpan<byte> ephemeralPrivate,
          ReadOnlySpan<byte> recipientPublic,
          ReadOnlySpan<byte> algorithmId,     // UTF-8 of "ECDH-1PU+A256KW", etc.
          ReadOnlySpan<byte> apu,
          ReadOnlySpan<byte> apv,
          ReadOnlySpan<byte> aeadTag,         // empty for outer wrap; AEAD tag for 1PU
          int keyDataLen);
  }
  ```
  Internally: `Ze = DeriveSharedSecret(curve, ephemeralPrivate, recipientPublic)`; `Zs = DeriveSharedSecret(curve, senderPrivate, recipientPublic)`; `Z = Ze ‖ Zs`; `suppPubInfo = BE(keyDataLen*8) ‖ aeadTag`; call `NetDid.Core.Crypto.Kdf.ConcatKdf.DeriveKey(Z, algId, apu, apv, suppPubInfo, default, keyDataLen)`.

### 3.6 ICryptoProvider

- [ ] `Crypto/ICryptoProvider.cs` — DIDComm-shaped surface. Methods accept JOSE algorithm strings:
  ```csharp
  internal interface ICryptoProvider
  {
      byte[] Sign(string joseAlg, ReadOnlySpan<byte> privateKey, ReadOnlySpan<byte> data);
      bool Verify(string joseAlg, ReadOnlySpan<byte> publicKey, ReadOnlySpan<byte> data, ReadOnlySpan<byte> signature);

      byte[] DeriveSharedSecret(string crv, ReadOnlySpan<byte> privateKey, ReadOnlySpan<byte> publicKey);

      // AEAD: encrypted by enc; routes to AES-CBC-HS512 / AES-GCM / XC20P.
      (byte[] Ciphertext, byte[] Tag) AeadEncrypt(string enc, ReadOnlySpan<byte> cek, ReadOnlySpan<byte> iv, ReadOnlySpan<byte> aad, ReadOnlySpan<byte> plaintext);
      byte[] AeadDecrypt(string enc, ReadOnlySpan<byte> cek, ReadOnlySpan<byte> iv, ReadOnlySpan<byte> aad, ReadOnlySpan<byte> ciphertext, ReadOnlySpan<byte> tag);

      // Key wrap (A256KW only in Phase 0; ECDH-ES+A256KW / ECDH-1PU+A256KW dispatch in Phase 2).
      byte[] KeyWrap(string alg, ReadOnlySpan<byte> kek, ReadOnlySpan<byte> cek);
      byte[] KeyUnwrap(string alg, ReadOnlySpan<byte> kek, ReadOnlySpan<byte> wrapped);
  }
  ```
  Internal-only in Phase 0; lifts to public in Phase 2 once the surface settles.
- [ ] `Crypto/DefaultCryptoProvider.cs`:
  - Constructor takes an optional `NetDid.Core.ICryptoProvider` (defaults to `new NetDid.Core.Crypto.DefaultCryptoProvider()`).
  - `Sign`/`Verify`: JOSE alg → `KeyType` + `EcdsaSignatureFormat` mapping table → delegate.
  - `DeriveSharedSecret`: `crv` string → `KeyType` → delegate.
  - `AeadEncrypt`/`AeadDecrypt`: dispatch to the right `IAead` instance.
  - `KeyWrap`/`KeyUnwrap`: `A256KW` only in Phase 0; throw for everything else.

## 5. Risks (revised)

- **R1. NSec on Windows CI.** Unchanged from the original plan — NSec ships native libsodium binaries via runtime IDs. Confirm restore on Windows. Mitigation: explicit `RuntimeIdentifiers` if needed.
- **R2. `Ecdh1PuKdfTests` synthetic vector.** We don't yet have a public 1PU test vector for Phase 0 (Appendix C.3's 1PU vector lands as a Phase 2 fixture). We hand-compute the expected output with an independent tool (Python `cryptography` or `pyjwt`) and pin it in the test. Risk: the hand-computed value is wrong. Mitigation: cross-check against the SICPA `didcomm-rust` 1PU code path using its existing unit-test fixtures; this is the same source we'll harvest in Phase 2 anyway.
- **R3. `Microsoft.IdentityModel.Tokens.JsonWebKey` vs. our own `Jwk` model.** If we adopt net-did's choice, we inherit its limitations (no easy unknown-header preservation). If we roll our own, we maintain a parallel converter. **Decision in §3.2 — roll our own with `AdditionalData` bag.**
- **R4. OpenTelemetry advisory drift.** Pinned `OpenTelemetry.Api 1.15.3` (current patched). Future security advisories will surface as NU1902 restore errors due to `warnaserror`; bump as they appear.
