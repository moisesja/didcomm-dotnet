## [ ] #27 — single source of truth for the outbound SSRF policy (Low, correctness)
- **Root cause:** three independent `OutboundEndpointPolicy` instances — `DidCommOptions` (facade
  pre-send `Validate`), `HttpTransportOptions` (connect-time pin), `WebSocketTransportOptions` (connect-time
  pin) — each defaulting to its own `new()`. Configuring one doesn't affect the others (fails closed —
  stricter wins — but confusing: allowlisting a host in `DidCommOptions` is still blocked by the transport
  default).
- **Fix (inherit-when-unset):** make the two transport policies **nullable** (`OutboundEndpointPolicy?`,
  default `null` = "inherit the core policy"), and resolve at connect time with fallback to
  `DidCommOptions.OutboundEndpointPolicy`:
  - HTTP (`HttpDidCommBuilderExtensions` ConnectCallback): `httpOpts.OutboundEndpointPolicy ??
    sp.GetService<IOptions<DidCommOptions>>()?.Value.OutboundEndpointPolicy ?? new()`.
  - WS (`WebSocketDidCommTransport` ctor): add an optional `IOptions<DidCommOptions>?` param;
    `_options.OutboundEndpointPolicy ?? core?.Value.OutboundEndpointPolicy ?? new()`.
  Net: by default all three resolve to `DidCommOptions.OutboundEndpointPolicy` (one source); an explicit
  transport policy still overrides. Minor public-API change (nullable property) — noted in CHANGELOG.
- **Tests:** with a custom `DidCommOptions.OutboundEndpointPolicy` and unset transport policy, the
  transport guard uses the core policy (e.g. an allowlisted host passes the connect guard); an explicit
  transport policy still wins.

## [ ] #31 — FR-THR-04 rule 3 (Low, spec) — scoped bookkeeping + honest docs
- **Root cause:** `AckLoopGuard` remarks claim rule 3 is "enforced in the dispatcher using IsPureAck +
  thread bookkeeping," but no bookkeeping exists; the dispatcher only does a rule-2-shaped check.
- **Fix:** real (dispatcher-scoped) rule-3 bookkeeping:
  - Add `ThreadState.AckRequested` (bool).
  - `ProtocolDispatcher`: fetch the thread early; **rule 3** = `IsPureAck(inbound) && thread.AckRequested`
    ⇒ drop (`DroppedAsAckLoop`) + clear the flag (consume the awaited ACK, don't dispatch → no loop).
    When the dispatcher produces a reply that `RequestsAck`, set `thread.AckRequested = true`.
  - Keep the existing rule-2-defensive drop (pure-ACK that also requests-ACK).
  - **Correct the `AckLoopGuard` remarks** to describe exactly what's enforced, and document that ACK
    requests sent OUTSIDE the dispatcher (facade-direct `SendAsync`) are the application's responsibility
    (the dispatcher only tracks ACK requests it emits).
- **Tests:** dispatcher emits an ack-requesting reply on a thread → a subsequent pure-ACK on that thread is
  dropped (rule 3) and the flag cleared; a pure-ACK with no prior request is NOT dropped by rule 3 (still
  dispatched); the rule-2-defensive drop still fires.

## [ ] #32 — JWS `typ` not validated on verify (Info, spec) — **close as documented, no code**
- The cited local `Jose/Signing/JwsParser.cs` / `JwsProtectedHeader.cs` were **deleted** in the JOSE
  delegation; the external `DataProofsDotnet.Jose.JwsParser` owns verification and does **not** surface
  `typ` on its `JwsParseResult`, so an in-repo check isn't possible without an upstream API change.
- #32 is **Info** and self-states "no functional change required for spec compliance": RFC 7515 §4.1.9
  makes `typ` advisory, FR-SIG-04 makes the signed `typ` a SHOULD on emit (and permits `JWM`) with no
  MUST to validate on receive, and the envelope kind is determined **structurally** (not by `typ`), so
  there is no kind-confusion risk. Validating `typ` would risk a non-compliant over-tightening (rejecting
  absent/`JWM`).
- **Disposition:** close #32 as spec-compliant / won't-fix-in-repo with this rationale; optionally note an
  upstream diagnostic-only enhancement (surface `typ` on `JwsParseResult`). _Flag for your confirmation —
  if you'd prefer an upstream change + lenient diagnostic, say so and I'll file it on dataproofs-dotnet._
