# Refactor v2: delegate the whole envelope layer to DataProofsDotnet.Jose (on NetCrypto 1.1.0)

**Date:** 2026-06-15
**Status:** PLAN — awaiting approval (no source/test/doc edits made yet)
**Supersedes:** `tasks/todo20260614T164909Z.md` (v1 assumed dataproofs had no JOSE; it now does).
**Driver:** NetCrypto **1.1.0** (closed all four gap issues #10/#11/#12 + dataproofs #6) and **`DataProofsDotnet.Jose`** now ships the full DIDComm JWE/JWS profile. This realizes architectural-path §5.6 ("didcomm uses dataproofs-dotnet for its JOSE needs rather than its own").

---

## 1. What changed since v1

- **NetCrypto 1.1.0** closed every gap I filed: `ExtractPublicKey` now guarantees on-curve validation (G2/#10); `IKeyStore.DeriveSharedSecretAsync` added (G3/#11); `NetCrypto.Base64Url` + AEAD `Key/Nonce/TagSizeBytes` added (G4/G5/#12).
- **`DataProofsDotnet.Jose`** (built on NetCrypto, verified the byte-identical *descendant* of didcomm's own `Jose/`+`Crypto/Kdf`) ships: `JweBuilder.BuildEcdh1PuA256Kw` (authcrypt, multi-recipient, `skid`), `BuildEcdhEsA256Kw` (anoncrypt), `JweParser.Parse` (+ `PeekRecipients`), `JwsBuilder.BuildJsonAsync`/`BuildCompactAsync`, `JwsParser.Parse`, `JoseCryptoProvider`, `Jwk`/`JwkConversion`, all DIDComm alg/enc constants, and A256CBC-HS512/A256GCM/XC20P.

**So v2 is a much bigger, much leaner refactor:** delete didcomm's **entire `Crypto/` folder and ~all of `Jose/`**, delegate envelope build/parse to `DataProofsDotnet.Jose`, and keep only the DIDComm message-layer.

## 3. Disposition

### DELETE — entire `Crypto/` folder (delegated)
`Crypto/Aead/{IAead,AesCbcHmacSha512,AesGcmAead,XChaCha20Poly1305Aead}.cs`, `Crypto/KeyWrap/AesKeyWrap.cs`, `Crypto/Kdf/{Ecdh1PuKdf,EcdhEsKdf}.cs`, `Crypto/KeyAgreement/EphemeralKeyPair.cs`, `Crypto/{ICryptoProvider,DefaultCryptoProvider}.cs` → all internal to `DataProofsDotnet.Jose` (JweBuilder/JweParser/JoseCryptoProvider). Byte-equivalence of AEAD/KW/KDF/apu/apv/suppPubInfo **confirmed** against the DataProofs source.

### DELETE — most of `Jose/` (delegated)
`Jose/JwkConversion.cs`, `Jose/Encryption/{ApuComputer,ApvComputer,RecipientWrap,JweProtectedHeader,JweBuilder,JweParser}.cs`, `Jose/Signing/{JwsProtectedHeader,JwsBuilder,JwsParser}.cs`, `Jose/Jwk.cs` → `DataProofsDotnet.Jose.*`. `Jwk` becomes a `global using Jwk = DataProofsDotnet.Jose.Jwk;` (shape-compatible incl. the FR-MSG-15 extension-data round-trip).

### KEEP — DIDComm message-layer (no foundation home)
- `Composition/EnvelopeWriter.cs` + `EnvelopeReader.cs` — legal-composition state machine + media-type stamping (FR-ENV-01..07); DataProofs treats `typ` as opaque and knows nothing of DIDComm nesting.
- `Jose/EnvelopeDetector.cs` + `EnvelopeKind.cs` — structural envelope sniff (FR-API-03).
- `Consistency/AddressingConsistency.cs` + `DidSubject.cs` — FR-CONSIST-01/02/03/05/06 (operate on the plaintext `Message` + net-did resolution; DataProofs only supplies the kids).
- `Facade/DidCommClient.cs` — FR-ENC-19 recipient defaulting, FR-ENC-04 curve negotiation (`PickCommonCurve`), `skid` selection, FR-ENC-09 user-facing enc rejection, FR-CONSIST-06 predicate, media-type/expiry.
- `Protocols/Rotation/{FromPriorBuilder,FromPriorValidator}.cs`, `Protocols/Routing/{ForwardWrapper,ForwardProcessor}.cs`, `Resolution/NetDidKeyService.cs` + `DidKeyServiceLookups.cs`, `Secrets/*`, `Json/DeterministicJsonWriter.cs` (now produces the canonical inner-Message bytes that become the JWS payload).
- **Trimmed** `Jose/Base64Url.cs` (BCL shim, used by OOB/forward/from_prior), `Jose/JoseAlgorithms.cs` (curve constants + `A256CbcHs512` referenced by facade/routing), `Crypto/KeyAgreement/KeyTypeMapper.cs` (crv→JWS-alg + crv-supported-per-relationship predicates used by `NetDidKeyService`/from_prior). Keeping these avoids re-pointing message-layer call-sites and a `NetCrypto.KeyType` leak.

## 4. Rewire (ordered)

1. **References** (§2) first, so it builds against the new surface.
2. **Jwk identity:** `global using Jwk = DataProofsDotnet.Jose.Jwk;`, delete `Jose/Jwk.cs`. Facade/secrets/resolver surfaces now speak DataProofs' `Jwk` → zero conversion at the JOSE boundary.
3. **Delete** the files in §3; trim `JoseAlgorithms`/`KeyTypeMapper`.
4. **Pack (`EnvelopeWriter`):** `PackAnoncrypt → JweBuilder.BuildEcdhEsA256Kw(inner, recipients, enc, jose, typ: MediaTypes.Encrypted)`; `PackAuthcrypt → JweBuilder.BuildEcdh1PuA256Kw(inner, recipients, senderPrivJwk, skid, enc, jose, typ)`; `anoncrypt(authcrypt)` outer = a second `BuildEcdhEsA256Kw`. One `var jose = new JoseCryptoProvider();` threaded from the facade.
5. **Pack signed:** keep FR-SIG-06 `to`-present guard locally; serialize `Message` to canonical bytes (retained `DeterministicJsonWriter`); map signer `Jwk → JwsSigner`; `JwsBuilder.BuildJsonAsync(payloadBytes, signers, typ: MediaTypes.Signed, ct)`.
6. **Unpack encrypted (`EnvelopeReader`):** `JweParser.Parse(packed, recipientResolver, senderResolver, jose)`; consume `JweParseResult.{Plaintext,Algorithm,ContentEncryption,RecipientKid,AllRecipientKids,SenderKid,IsAuthenticated}` as today.
7. **Unpack signed (`EnvelopeReader`):** `JwsParser.Parse(packed, resolveSignerPublicJwk, jose)` → returns **`PayloadBytes`** (not a `Message`). **EnvelopeReader must now** deserialize `PayloadBytes → Message` (with the malformed-input guard) **and run the two checks that previously lived inside didcomm's JwsParser** (see §5).
8. **Resolver adapters:** implement `IJweRecipientKeyResolver` over `IInternalSecretsLookup`, `IJweSenderKeyResolver` over `IDidKeyService`(keyAgreement), and the JWS `Func<string,Jwk?>` over `IDidKeyService`(authentication). Keep the async→sync bridging (DataProofs resolvers are sync).
9. **from_prior (FR-ROT-01):** sign via `JwsBuilder.BuildCompactAsync` + `JwsSigner` (because `IJoseCryptoProvider` has **no `Sign`**); keep FR-ROT-01/02/freshness claim checks + `FromPriorValidator` verify via `JwsParser.ParseCompact` + DataProofs `JwkConversion.ExtractPublicKey`. **Validate byte output** of from_prior (its bespoke header/claims ordering differs from DataProofs canonicalization — see §6 R3).
10. **`NetDidKeyService.ProjectMethod`:** replace the removed `Multibase/Multicodec/KeyTypeExtensions/NetDidJwkConverter` chain with `DataProofsDotnet.Jose.JwkConversion.FromMultikey(publicKeyMultibase, kid)`. Keep `IsSupported(crv, relationship)` on the trimmed local mapper.
11. **`DidComm.Adapters.NetDid`** (separate package): `NetDidKeyStoreSecretsResolver` uses `NetDid.Core.IKeyStore`+`KeyInfo.KeyType` — **removed in 2.0.0**. Rewrite against `NetCrypto.IKeyStore` (now has `DeriveSharedSecretAsync`, so it can even support HSM-bound decryption) or retire. Does not block `DidComm.Core`.

## 5. Re-homed security checks (MUST NOT regress) — HIGH

Because `JwsParser` now returns bytes, these move **up into `EnvelopeReader`** (verified to currently live inside `JwsParser.cs:117-122` / `EnvelopeReader.cs:145`):
- **FR-CONSIST-03** signed `from` ↔ signer `kid`: `AddressingConsistency.CheckSignedFromMatchesSignerKid(message.From, JwsParseResult.SignerKid)` after deserializing the payload.
- **FR-SIG-06** inner signed JWM under encrypt MUST carry `to`: re-assert on the deserialized inner `Message`.
- Unchanged but re-confirmed at their slot-in points: FR-CONSIST-01 (`from↔SenderKid`), -02 (`RecipientKid∈to`), -05 (authcrypt(sign) inner↔outer), -06 (resolver-backed authorization). `IsAuthenticated` still drives the auth/anon metadata.

## 6. Gaps / decisions to surface

| # | Sev | Item | Recommendation |
|---|---|---|---|
| **R1** | High | FR-CONSIST-03 + FR-SIG-06 re-homing (§5) | In-scope rewire step, not a foundation gap. Cover with the existing forged-signer + missing-`to` negative tests. |
| **R2** | Low→Med | **ES512 / P-521 JWS signing** rejected by DataProofs `JwsSigner` (v1 scope). didcomm maps P-521→ES512 today. | No DIDComm vector needs ES512 (FR-SIG-01 MUST = EdDSA/ES256/ES256K). **Recommend:** accept the narrowing + document it; **optionally file a low-pri dataproofs issue** to add ES512. (P-521 *key agreement* is unaffected — only JWS *signing*.) |
| **R3** | Med | **from_prior** byte output via `JwsBuilder.BuildCompactAsync` may differ from the current bespoke `{alg,kid,typ:JWT}`+lexicographic-claims encoder. | Validate round-trip + any stored from_prior vector; if exact bytes matter, keep a thin local compact-JWS encoder using a minimal signing primitive. |
| **R4** | Low | FR-ENC-13 **apv relaxation**: DataProofs re-verifies `apv` only when present; didcomm's ancestor required it. Real DIDComm envelopes always set `apv` (byte-identical), but an `apv`-less envelope would now be accepted. | Optional: add a strict "apv present for DIDComm" assert on unpack to preserve exact reject behavior. |

## 7. Validation (the equivalence gate)
- Round-trip the **SICPA/askar interop vectors** (authcrypt 1PU + anoncrypt + signed) through the rewired pack/unpack: assert identical wire bytes on pack and successful cross-decrypt on unpack. This is the only thing that fully proves the byte-equivalence the analysis asserts (apu/apv/1PU/suppPubInfo/AAD/canonicalization).
- Negative paths: off-curve epk (FR-ENC-03, now in NetCrypto), forged self-consistent JWK, forged signer (FR-CONSIST-03 from EnvelopeReader), missing-`to` inner (FR-SIG-06), authcrypt-with-GCM rejected (FR-ENC-09), `RecipientKid∉to`, mismatched `from↔skid`.
- Full `DidComm.Core.Tests` + `DidComm.InteropTests` green; build confirms the kept message-layer compiles against the new surface.

## Review section (completed 2026-06-16)

**Outcome: done and validated — 574/574 tests green (475 unit + 99 interop) under `warnaserror`.**

What landed vs. the plan:
- Deleted the entire `Crypto/` folder and most of `Jose/` (AEAD, KDF, key-wrap, JWE/JWS build+parse,
  apu/apv, JWK conversion, the protected-header DTOs). Kept `Base64Url`, `EnvelopeDetector`,
  `EnvelopeKind`, a trimmed `JoseAlgorithms` (string constants) and `KeyTypeMapper`
  (`ToJwsAlgorithm` + curve predicates), plus one new `JwsSignerFactory`.
- Pack/unpack delegate to `DataProofsDotnet.Jose` (`JweBuilder`/`JweParser`/`JwsBuilder`/`JwsParser`,
  `JoseCryptoProvider`). Pack path went async. `Jwk` is now `DataProofsDotnet.Jose.Jwk` (global
  alias). The internal secret/sender lookups **inherit** DataProofs' resolver interfaces, so no
  adapter shims were needed (cleaner than the plan's "thin adapter").
- **R1 (re-homed checks):** FR-CONSIST-03 and FR-SIG-06 moved into `EnvelopeReader`'s plaintext
  branch (the JWS parser returns bytes, not a Message). Covered by the existing forged-signer /
  missing-`to` negatives — all green.
- **R5 (adapter):** `NetDidKeyStoreSecretsResolver` rewritten against `NetCrypto.IKeyStore`.
- **Deviation from the plan:** a direct `NetCrypto` reference *was* needed after all — DataProofs'
  JWS layer signs through `ISigner` (not raw bytes), so `JwsSignerFactory` adapts a private JWK into
  a `NetCrypto.KeyPairSigner`. This is the only direct NetCrypto touch; everything else is DataProofs.
- **R2 (ES512/P-521 JWS signing):** accepted the narrowing (no DIDComm vector needs ES512). A P-521
  signer JWK surfaces a clear `NotSupportedException`.
- **dataproofs-dotnet#10 (discovered during validation):** DataProofs' `JwsParser` dropped the
  verified signer kid when it sat in the JWS *unprotected* header (where DIDComm v2.1 puts it),
  failing 5 spec vectors. Filed with a full security analysis; fixed upstream in **1.0.1**; all 5
  went green. didcomm's consistency checks were never weakened.
- Dependency reference for DataProofsDotnet.Jose ended on a NuGet `PackageReference` (1.0.1) per the
  user's later instruction (started as a local project reference while the prefix reservation was
  pending).

Net result: ~435+ LOC of hand-rolled, security-sensitive crypto deleted from this repo; NSec direct
dependency removed; NetDid demoted to resolution-only. Docs updated (PRD §3.1/§3.2, CHANGELOG,
lessons L-022/L-023).
