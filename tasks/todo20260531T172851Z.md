## Workstream A ‚Äî Out-of-Band 2.0 (FR-OOB-01..05)

**Design decisions (please veto any in review):**
- **No `IProtocolHandler`.** Per spec the OOB protocol "consists in a single message sent by the sender," delivered out-of-band (URL/QR) and decoded locally; the recipient then *initiates* a new protocol with `pthid = invitation.id`. There is no inbound invitation to dispatch through `ProtocolDispatcher`, so OOB ships as a **static API + types + URL helpers + AspNetCore serving endpoint**, not a registry handler. (Matches the explore finding.)
- **`goal_code`/`goal`/`accept` go in `body`** (reuse `MessageBuilder.WithBody(JsonObject)`); `attachments` via `WithAttachment(...)`.
- **Reuse, don't rebuild:** `EnvelopeWriter.PackPlaintext(Message)` already emits compact (whitespace-free) JSON; `Jose.Base64Url` (internal, **same `DidComm.Core` assembly** ‚Äî no public change needed) does encode/decode; `MessageTypeUri` validates the type.
- **FR-OOB-02 "byte-for-byte" caveat:** JSON object key order is **not** spec-canonical (the spec example orders `type,id,from,body,attachments`; our serializer uses property-declaration order). True byte-for-byte *encode* equality with the spec sample is therefore not guaranteed. We satisfy the requirement by (1) **decoding the spec's literal `_oob` example** and asserting the parsed invitation, and (2) **round-trip equality** (`FromUrl(ToUrl(x)) == x`). This caveat will be noted in the XML docs and CHANGELOG.

**New folder `src/DidComm.Core/Protocols/OutOfBand/`** (namespace `DidComm.Protocols.OutOfBand`), mirroring the TrustPing/DiscoverFeatures convention:
- `OutOfBand.cs` ‚Äî static API:
  - constants `ProtocolUri = "https://didcomm.org/out-of-band/2.0"`, `InvitationType = ".../invitation"`, body field names.
  - `CreateInvitation(string from, string? goal = null, string? goalCode = null, IEnumerable<string>? accept = null, IEnumerable<Attachment>? attachments = null, string? id = null)` ‚Üí `OutOfBandInvitation`.
  - `ToUrl(OutOfBandInvitation, string baseUrl)` ‚Üí `"{baseUrl}?_oob={b64url}"` (preserves existing query params on `baseUrl`).
  - `FromUrl(string url)` ‚Üí `OutOfBandInvitation` (extract `_oob`, base64url-decode, deserialize+validate type).
  - `FromPlaintext(string json)` ‚Üí `OutOfBandInvitation` (for short-form GET bodies).
  - short-form helpers: `ToShortUrl(string baseUrl, string oobId)` ‚Üí `"{baseUrl}?_oobid={oobId}"`; `bool TryGetShortFormId(string url, out string oobId)`.
  - web_redirect (FR-OOB-05): `WebRedirect? ReadWebRedirect(Message)` and `Message WithWebRedirect(MessageBuilder, WebRedirect)` reading/writing the top-level `web_redirect` via `Message.AdditionalHeaders` (`[JsonExtensionData]`).
- `OutOfBandInvitation.cs` ‚Äî sealed type wrapping the invitation `Message`; surfaces `Id`, `From`, `Goal`, `GoalCode`, `Accept`, `Attachments`, and `Message Message { get; }`.
- `WebRedirect.cs` ‚Äî `sealed record WebRedirect(string Status, string RedirectUrl)`.
- `IOobInvitationStore.cs` + `InMemoryOobInvitationStore.cs` ‚Äî mirror `Threading/InMemoryThreadStateStore.cs` (`ConcurrentDictionary<string,string>`, `Store/Retrieve/Remove`); demo/test grade, for FR-OOB-04.

**AspNetCore (FR-OOB-04 serving side):** add to `src/DidComm.AspNetCore/DidCommEndpointRouteBuilderExtensions.cs`:
- `MapDidCommOobEndpoint(this IEndpointRouteBuilder, string pattern, IOobInvitationStore store)` ‚Äî `MapGet` reading `_oobid`; 200 + plaintext invitation (`Content-Type: application/didcomm-plain+json`) when found, 404 when not, 400 on missing/empty id. Matches existing endpoint conventions.

Demonstrates build ‚Üí `ToUrl` ‚Üí `FromUrl` round trip, short-form store + GET retrieval via the endpoint, and `pthid = invitation.Id` correlation on a response. ## Workstream B ‚Äî NuGet publishing pipeline (NFR-08)

Most metadata already exists in `Directory.Build.props` (Authors, Apache-2.0, RepositoryUrl, tags, `IncludeSymbols`, `snupkg`, SourceLink) and all 6 `src/` projects carry `PackageId`/`Version=$(DidCommVersion)`/`Description`. Gaps to close:

1. **README packaging fix (blocking for `dotnet pack`).** `PackageReadmeFile=README.md` is set but no project includes the file ‚Üí `dotnet pack` fails (NU5039). Add to `Directory.Build.props`, guarded so it only applies to packable projects:
   ```xml
   <ItemGroup Condition="'$(IsPackable)' != 'false'">
     <None Include="$(MSBuildThisFileDirectory)README.md" Pack="true" PackagePath="\" Visible="false" />
   </ItemGroup>
   ```
2. **Tag-driven version.** Keep `DidCommVersion=0.1.0` as the dev default; the release workflow overrides it from the pushed tag (`-p:DidCommVersion=${TAG#v}`). No new dependency (avoids MinVer/Nerdbank per the dependency-light preference).
3. **`.github/workflows/release.yml`** ‚Äî trigger on `push: tags: ['v*']`; single `ubuntu-latest` job mirroring `ci.yml` setup (checkout, `setup-dotnet` via `global.json`); `dotnet build -c Release /warnaserror` ‚Üí `dotnet test -c Release` (gate) ‚Üí `dotnet pack -c Release -p:DidCommVersion=<tag> -o artifacts` ‚Üí `dotnet nuget push artifacts/*.nupkg --api-key ${{ secrets.NUGET_API_KEY }} --source https://api.nuget.org/v3/index.json --skip-duplicate` (snupkg pushed alongside). `permissions: contents: read`; upload `artifacts/` as a workflow artifact.
4. **README.md** ‚Äî replace the "No NuGet packages have been published yet" line with an install/quickstart section once the workflow exists (packages list + `dotnet add package DidComm.Core`).
5. **Prerequisite I cannot do (call out for you):** add the `NUGET_API_KEY` repository secret (nuget.org API key), and push the first `v0.1.0` tag to trigger publish. I'll document both in the PR. (OIDC trusted publishing is a later option.)
