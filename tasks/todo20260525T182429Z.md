# Phase 6 — Protocols, Cross-Message Concerns, Live Interop, Samples, Release

**PRD §12 Phase 6.** Closes: **FR-PROTO-01..11/11a, FR-OOB-01..05, FR-THR-01..04,
FR-PROF-01..02, FR-I18N-01..04, FR-IX-04..09, FR-DX-01..09, NFR-02/05/06/07/09.**

This is the largest phase in the PRD (~30 FR IDs spanning code, samples, CI, docs
and release). I'm proposing it as **eight sequential checkpoints**, each a feature
branch and its own PR, mirroring the cadence Phases 4/5 used (A/B/C/D… → one PR
per checkpoint, merged before the next starts). This keeps each PR reviewable,
keeps `main` green at every step, and lets you stop / re-plan between checkpoints
without rework.

## Starting state (audited 2026-05-25)

- Phases 0–5 complete; transports green; 364 unit + 69 interop tests passing.
- `MessageTypeUri` + `ProtocolVersion` parser/semver already exist (FR-PROTO-01/02 done).

- Cookbook has 7 sections (K, N, O, P, Q, R, AA); **missing A–J, L, M, S–Z, V, BB, Y, Z** and **all 10 standalone sample apps** (only the cookbook + `_shared` exist).

## Cross-cutting rules

- One feature branch per checkpoint: `feat/phase-6-<slug>`. Never commit to `main`.
- PRD updates only when the implementation diverges from the spec text (per CLAUDE.md "PRD is the target").
- Every new public type ships XML docs + an `<example>` link to the demonstrating Cookbook section (FR-DX-08, NFR-02).
- `dotnet build /warnaserror` clean + `dotnet test DidComm.sln` green is the bar to merge each checkpoint.
- After each merged checkpoint: CHANGELOG entry + lessons appended.

## Checkpoint 6.2 — `IProtocolHandler` registry + 5 built-in handlers (FR-PROTO-03..11/11a)

**Branch:** `feat/phase-6-protocols`

**Adds:**
- `IProtocolHandler { string ProtocolUri; Task<Message?> HandleAsync(Message, ProtocolContext, CancellationToken); }`
- `ProtocolHandlerRegistry` — dispatch by PIURI prefix (FR-PROTO-03); same-major/older-minor compat (FR-PROTO-02).
- `ProtocolContext` — exposes the `UnpackResult`, the current `ThreadState`, and an `IDidCommClient` for replies.
- Built-in handlers (each is its own file under `Protocols/<Name>/`):
  - **TrustPing 2.0** — `TrustPing.CreatePing(...)`, `TrustPingHandler` (FR-PROTO-04). Auto-replies `ping-response` with `thid = ping.id` when `response_requested` (default true).
  - **DiscoverFeatures 2.0** — `DiscoverFeatures.CreateQuery(...)`, `DiscoverFeaturesHandler` (FR-PROTO-05). Feature-types `protocol`, `goal-code`, `header`, `constraint`; `*` wildcard + prefix match; `max_receive_bytes` constraint round-trips through `DidCommOptions.MaxReceiveBytes`.
  - **Empty 1.0** — `Message.Empty()` static + handler (FR-PROTO-06). Used by ACK-only replies.
  - **ProblemReport 2.0** — `ProblemReport.Create(code, pthid, comment?, args?, escalateTo?, ack?)` + `ProblemCode` taxonomy (sorter `e/w`; scope `p/m/<state>`; descriptors `trust[.crypto]`, `xfer`, `did`, `msg`, `me[.res…]`, `req[.time]`, `legal`) — prefix matching (FR-PROTO-07/08); `{n}` interpolation (missing→`?`, extras appended); warning→error escalation (FR-PROTO-09); per-thread max-error cascade guard emitting `e.p.req.max-errors-exceeded` (FR-PROTO-10).
  - **Trace 2.0** — `TraceHandler` **off by default** (FR-PROTO-11/11a). Requires explicit opt-in + safeguards; emits `trace_report` POST when enabled.

**Plumbing:**
- `DidCommBuilder.AddProtocol<T>()` + `AddBuiltInProtocols()`.
- New `MapDidCommEndpoint` / `MapDidCommWebSocket` overload that takes no callback and dispatches through the registry — falls through to a registry-aware default `onReceive` that runs the dispatched reply back through `DidCommClient.SendAsync` when the handler returns a non-null `Message`. Existing inline-callback overload retained for tests and custom flows.
- ACK-loop and thread-scoped accept-lang integration (depends on 6.1).

**Tests:**
- Per-handler unit tests (positive + spec-example).
- ProblemReport interpolation table (missing arg = `?`, extras appended).
- Cascade-guard breach stops further replies on that `thid`.
- Registry dispatch by MTURI ignoring case/punctuation; semver compat.

**Cookbook (new):**
- S — Trust Ping liveness.
- T — Discover Features handshake (incl. `max_receive_bytes`).
- U — Report a problem (codes + interpolation + escalation).
- W — Empty message (header-only ACK).
- X — Custom `IProtocolHandler` (`lets_do_lunch`).

**Exit:** all five spec protocols pass unit tests; ProblemReport handles interpolation/escalation/cascade per FR-PROTO-07..10; Trace silent by default (FR-PROTO-11a); cookbook sections S/T/U/W/X run end-to-end.

## Decisions (locked 2026-05-25)

1. **8-checkpoint cadence approved.** Smaller cuts allowed inside a checkpoint if a single PR grows unreviewable.
2. **Start at 6.1.**
3. **Live interop (6.7) = on-demand only.** No nightly. Use `workflow_dispatch` so the harness runs only when manually triggered from the Actions tab or on release-tag pushes. No scheduled cron job, no Python/JVM costs on PR or nightly CI.
4. **Trust Ping defaults: match `sicpa-dlab/didcomm-python`.** `response_requested` defaults true; thread linkage `thid = ping.id`.
5. **Samples (6.5) split into two PRs:**
   - **6.5a** — samples 01-Quickstart, 03-EnvelopesAndMessages, 04-MediatorAgent + README quickstart.
   - **6.5b** — samples 05-WebSocketChat, 06-OutOfBand, 07-ProblemsAndProtocols, 08-Extensibility, 09-NetDidIntegration, 10-ProfilesAndI18n + remaining cookbook sections + **API-coverage CI gate** (FR-DX-01).
   - The API-coverage gate lands in 6.5b (after all samples exist) so it doesn't false-fail on partial 6.5a state.
