## SSRF hardening — when does it land?

Per PRD investigation:

- **FR-PROTO-11** ("Trace 2.0: support honoring the `trace` header by POSTing a `trace_report`…") is a **MAY** requirement, bundled into Phase 6's general scope ([docs/didcomm-dotnet_PRD.md:446](docs/didcomm-dotnet_PRD.md)).
- **FR-PROTO-11a** (off-by-default) is **MUST** — that's what this PR delivers.
- Phase 6's "Closes" list ([PRD line 608](docs/didcomm-dotnet_PRD.md)) names FR-PROTO-11 + FR-PROTO-11a together; the exit criterion says "Trace (off by default) all pass unit tests" — does **not** explicitly require the HTTP POST integration to ship for Phase 6 to close.
- **No specific sub-phase or follow-up PR is currently scheduled for Trace HTTP POST** integration. Task list shows Phase 6.3 (OOB) next, then live interop / OTel / samples-DX / release.

So the honest answer: **SSRF hardening isn't on the schedule**. It rides with the HTTP POST integration, which is a deferred MAY-tier item with no committed owner / sub-phase. Either (a) schedule a Phase 6.2d explicitly for "Trace HTTP POST + SSRF guard", or (b) accept the library may ship 1.0 with FR-PROTO-11a only (decision logic) and FR-PROTO-11 (POST) deferred past 1.0.

For this PR's review-fixup: **strike the fabricated loopback bullet** and add a clear `// TODO(FR-PROTO-11):` marker pointing at the future POST integration as the place where the real SSRF hardening lands. I'd recommend separately raising the scheduling question with the user (or flagging a follow-up task), but that's out of scope for this fix-up PR.

**2. Fix `JsonNode.AsValue()` crash on malformed bodies.**

File: [ProblemReport.cs](src/DidComm.Core/Protocols/ProblemReport/ProblemReport.cs)

```csharp

return node is JsonValue v && v.TryGetValue<string>(out var s) ? s : null;
```

**3. Allowlist matching: use `Uri.AbsoluteUri`, not `Uri.ToString()`.**

File: [TraceObserver.cs](src/DidComm.Core/Protocols/Trace/TraceObserver.cs)

Change:

```csharp
if (!options.AllowedReportingUris.Contains(parsed.ToString())) return false;
```

to:

```csharp
if (!options.AllowedReportingUris.Contains(parsed.AbsoluteUri)) return false;
```

Update `TraceOptions.AllowedReportingUris`'s XML doc to call out that entries are matched against the URI's **canonical** (percent-encoded) form so operators can put canonical URIs in the allowlist and expect them to match peer-supplied `report_uri` values that may be encoded differently.

**5. `ProblemReportHandler` — validate `From`/`To` *before* incrementing `ErrorCount`.**

File: [ProblemReportHandler.cs](src/DidComm.Core/Protocols/ProblemReport/ProblemReportHandler.cs)

Currently the From/To check sits inside the `count == threshold + 1` trip branch. If a malformed (no-From or no-To) report arrives at exactly the trip point, the cascade-stop is silently dropped and the next valid message falls into the `> threshold + 1` silent branch — the cascade-stop is **lost forever** for that pthid.

Fix: move the From/To guard to the top, immediately after the `code.IsError` check. A message that can't form a valid reply is a malformed report; we drop it without incrementing. Remove the duplicate check at the trip point.

Add a unit test: build a problem-report with empty `From`, dispatch threshold-many times, then dispatch one valid report — the valid one should be the count=1 case (no trip), proving the malformed ones weren't counted.
