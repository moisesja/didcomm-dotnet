## Why split 6.2 into three PRs

A single PR landing all five handlers + dispatcher + cookbook sections + ACK
rules 1/3 + cascade guard would be ~2,500 LOC across ~35 files and unreviewable
in one sitting (the size of the merged PR #8 with the SSRF rider was already at
the comfort ceiling). The split below keeps each PR around the size of 6.1.
The framework lands first so subsequent PRs only add handlers; the cookbook
sections in 6.2a/b/c only depend on the framework + their own handler, so each
sub-PR is independently runnable.

## Cross-cutting design (applies to all three sub-PRs)

### `IProtocolHandler`
```csharp
public interface IProtocolHandler
{
    /// PIURI without trailing slash, e.g. "https://didcomm.org/trust-ping/2.0".
    string ProtocolUri { get; }
    /// Returns a reply Message? for the dispatcher to pack-and-send, or null for none.
    Task<Message?> HandleAsync(Message message, ProtocolContext context, CancellationToken ct);
}
```
- `ProtocolHandlerRegistry`: `Register(IProtocolHandler)`, `TryResolve(string mturi, out IProtocolHandler)`. Internally indexes by PIURI prefix using `MessageTypeUri.Parse`; selects by same-major + nearest-≤ minor (FR-PROTO-02).
- `ProtocolContext`: read-only carrier — `UnpackResult Received`, `ThreadState Thread`, `IDidCommClient Client` (for replies + dispatch), `DidCommOptions Options`.
- Reply pipeline: if a handler returns `Message? reply ≠ null`, the dispatcher invokes `AckLoopGuard.IsSafeToSend(reply)` (FR-THR-04 rule 2); on `false` it throws `InvalidOperationException` to surface the bug in the handler.
- The dispatcher pre-filter (before calling `HandleAsync`):
  - **FR-THR-04 rule 3**: when the incoming is a pure ACK (`IsPureAck`) and its `ack[]` entries intersect the receiver's `ThreadState.OutboundPleaseAckMessageIds` AND those were emitted by a pure-ACK of ours, drop silently.
  - **FR-THR-04 rule 1**: a single incoming triggers at most one ACK reply per handler invocation — natural property of "handler returns one Message?".

### `ThreadState` additions (6.2a)
- `int ErrorCount` — FR-PROTO-10 cascade guard counter (incremented by ProblemReport handler in 6.2c; declared in 6.2a so the field is stable).
- `HashSet<string> OutboundPleaseAckMessageIds` — FR-THR-04 rule 3 bookkeeping. Populated when this side sends `please_ack`; consulted on receipt of a pure ACK.

### Endpoint overload — single shared design
```csharp
public static IEndpointConventionBuilder MapDidCommEndpoint(
    this IEndpointRouteBuilder endpoints,
    string pattern,
    DidCommReceiveOptions? options = null); // no onReceive — uses registry
```
- New overload: dispatches the unpacked message through `ProtocolHandlerRegistry`; if a reply is produced, packs (authcrypt back to sender) and either (a) for HTTP, returns 202 (does not write reply on the request response — DIDComm is one-way per FR-TRN-10), or (b) for WebSocket, sends the packed bytes on the same socket only if `DidCommReceiveOptions.AllowSameSocketReplies = true` (default `false` — same-socket replies violate FR-TRN-10 by default; opt-in).
- Existing inline-callback overload retained verbatim — tests and custom flows keep working.

### Lessons we already know (don't re-discover)
- L-003: every `<param>` needs a doc tag (warnaserror).
- L-004: alias when consuming `NetDid.Core.ICryptoProvider` from `DidComm.*`.
- L-006: when promoting types to public, walk the transitive surface.
- L-008: argument-shape guards go before resolver calls.
- L-014: namespace-vs-class shadowing (`DidComm.Profiles.Profiles`) needs a `using XxxConst = …;` alias in consumers.

- `src/DidComm.Core/Protocols/ProblemReport/ProblemReportOptions.cs` —
  `int CascadeThreshold = 5` (configurable).

  silently ignores `trace` header (no report); when `Enabled = true`, posts
  a `trace_report` to the `trace.report_uri` only if it's on the allowlist
