# SSRF fix — egress validation for outbound DIDComm sends (added to PR #8)

## Context

Security review (full `main` scan) found one HIGH-confidence vulnerability:

**SSRF via attacker-controlled DID `serviceEndpoint` host.** A counterparty's DID-document
`service[].serviceEndpoint.uri` is parsed verbatim (`ServiceEndpointParser.ProjectMap`) and flows —
with no host/IP validation anywhere — through `DidCommClient.SendAsync` (resolved-from-DID branch)
into an outbound HTTP/WebSocket POST of the packed envelope. The only constraint is a default
*scheme* check (`https`/`wss`); the **host is fully attacker-controlled**. With the default
self-asserted `did:key`/`did:peer` methods, an attacker hands the victim a DID whose endpoint is
`https://169.254.169.254/...` (cloud metadata) or an internal host, and the victim's server makes
the request. The HTTP 307-redirect follow path re-checks only the scheme of `Location`, not the host.

Confirmed by an independent filter sub-agent: REAL, confidence 8/10, in scope (host-control SSRF).

## Threat model / design decisions

- `SendOptions.ServiceEndpointOverride` is **caller-supplied = trusted** (analogous to a CLI flag).
  It stays UNvalidated so dev/test/loopback overrides keep working. The fix targets only the
  **resolved-from-DID** endpoint path, which is the untrusted data flow.
- Default policy = **block private / loopback / link-local / ULA / CGNAT / metadata** destinations,
  with an opt-out (`BlockPrivateNetworks = false`) and an `AllowedHosts` allowlist for legitimate
  internal mediators.
- DNS rebinding / redirect-to-internal are real secondary vectors → harden the transports at
  **connect time** (the IP the socket actually connects to), which also keeps existing unit tests
  offline (they swap the primary handler / inject a custom WS connect, bypassing production paths).

## Plan

### New files (Core)
1. `src/DidComm.Core/Transports/OutboundEndpointPolicy.cs`
   - `bool BlockPrivateNetworks { get; set; } = true`
   - `ISet<string> AllowedHosts { get; set; }` (OrdinalIgnoreCase) — hosts that bypass the block
   - `bool ResolveDnsNames { get; set; } = true` — resolve hostnames to catch internal names
2. `src/DidComm.Core/Transports/OutboundEndpointGuard.cs`
   - Built from an `OutboundEndpointPolicy` (+ injectable `Func<string, IPAddress[]>` DNS hook,
     default `Dns.GetHostAddresses`, for testability without network).
   - `void Validate(Uri endpoint)` → throws `TransportException` (scheme = endpoint.Scheme) when the
     destination is disallowed. Allowlist short-circuits. IP literal → classify directly; hostname →
     resolve (when enabled) and reject if **any** resolved address is private/reserved.
   - `static bool IsPrivateOrReserved(IPAddress)` → loopback, IPv4 link-local 169.254/16 (incl.
     metadata 169.254.169.254), private 10/8 + 172.16/12 + 192.168/16, CGNAT 100.64/10,
     unspecified 0.0.0.0/::, IPv6 loopback ::1, link-local fe80::/10, ULA fc00::/7, and
     IPv4-mapped-IPv6 unwrap before classifying.

### Edits (Core)
3. `src/DidComm.Core/Facade/DidCommOptions.cs` — add
   `public OutboundEndpointPolicy OutboundEndpointPolicy { get; set; } = new();`
4. `src/DidComm.Core/Facade/DidCommClient.cs` — in `SendAsync`, after resolving `endpointUri` from
   the DID (the `options.ServiceEndpointOverride is null` branch, after `Uri.TryCreate`), call
   `new OutboundEndpointGuard(_options.OutboundEndpointPolicy).Validate(endpointUri)` before building
   the `TransportRequest`. Override path remains untouched.

### Edits (transports — defense in depth, production paths only)
5. `src/DidComm.Transports.Http/HttpTransportOptions.cs` — add
   `public OutboundEndpointPolicy OutboundEndpointPolicy { get; set; } = new();`
6. `src/DidComm.Transports.Http/HttpDidCommBuilderExtensions.cs` — switch the named-client primary
   handler to `SocketsHttpHandler` with `AllowAutoRedirect = false` and a `ConnectCallback` that
   resolves the target host, drops blocked IPs via the guard, throws if none remain, and connects to
   an allowed IP. Covers initial connect + each manual 307 hop + DNS rebinding. (Unit tests call
   `ConfigurePrimaryHttpMessageHandler` with a stub, so this production handler is bypassed → tests
   stay offline and green.)
7. `src/DidComm.Transports.WebSocket/WebSocketTransportOptions.cs` — add the same
   `OutboundEndpointPolicy` property.
8. `src/DidComm.Transports.WebSocket/WebSocketDidCommTransport.cs` — in `DefaultConnect` (production
   path only; tests inject `options.Connect`), `guard.Validate(endpoint)` before connecting.

### Tests
9. `tests/DidComm.Core.Tests/Transports/OutboundEndpointGuardTests.cs` — table-driven: blocks
   127.0.0.1, ::1, 10.x, 172.16.x, 192.168.x, 169.254.169.254, fc00::, fe80::, 100.64.x,
   IPv4-mapped; allows a public IP; allowlist bypass; `BlockPrivateNetworks=false` bypass; hostname
   path via a stubbed DNS hook (internal-resolving name blocked, public-resolving name allowed).
10. `tests/DidComm.Core.Tests/Transports/DidCommClientSendAsyncTests.cs` — add: resolved endpoint at
    a private/metadata host throws `TransportException`; override to a private host still sends
    (trusted); `BlockPrivateNetworks=false` allows the resolved private host.

### Docs / process
11. `CHANGELOG.md` — add a Security entry describing the SSRF fix and the new policy knobs.
12. `docs/codebase-architecture.md` — note the egress guard if a transport/routing section exists
    (update only if it merits it).
13. `tasks/lessons.md` — capture the lesson (validate resolved/remote-derived URIs before egress).

### Verify
14. `dotnet build` the solution; `dotnet test` (Core.Tests + InteropTests). Confirm existing

15. Commit to `feat/phase-6-threading-i18n` and push so it lands in PR #8.

## Open scope question for the user
- **Recommended:** full scope above (Core client guard + HTTP/WS connect-time hardening).
- **Minimal:** Core client guard only (items 1–4, 9–10) — closes the reported finding at the choke
  point but leaves the 307-redirect / DNS-rebinding secondary vectors unaddressed.

## Review (completed)

Implemented the **full** scope (user-approved).

Added:
- `src/DidComm.Core/Transports/OutboundEndpointPolicy.cs` — `BlockPrivateNetworks` (default true),
  `AllowedHosts` (OrdinalIgnoreCase), `ResolveDnsNames` (default true).
- `src/DidComm.Core/Transports/OutboundEndpointGuard.cs` — `Validate(Uri)` pre-send gate +
  `ConnectAsync(DnsEndPoint, ct)` connect-time gate + static `IsPrivateOrReserved(IPAddress)`.
  Injectable DNS seam for offline tests. IPv4-mapped IPv6 unwrapped; blocks loopback / RFC 1918 /
  link-local+metadata / CGNAT / multicast+reserved / IPv6 link-local+site-local+multicast / ULA.

  path left trusted.
- `HttpTransportOptions` + `HttpDidCommBuilderExtensions` — named client now uses
  `SocketsHttpHandler` with `AllowAutoRedirect = false` and a `ConnectCallback` that vets + pins the
  TCP connection (covers initial connect, every 307 hop, and DNS rebinding).
- `WebSocketTransportOptions` + `WebSocketDidCommTransport` — vet the host on the default connect
  path (skipped when a custom `Connect` delegate is supplied).

Verification: full solution `dotnet test` green — **502 pre-existing + 8 new** (Core.Tests 435,

Residual notes:
- The pre-send WS `Validate` has a small resolve-then-connect TOCTOU window; the client-side guard
  plus the HTTP connect-time pin cover the primary vectors. A WS connect-time pin would require
  custom socket plumbing on `ClientWebSocket` and was deemed out of scope.
- Secure-by-default means loopback/private HTTP+WS targets now require `BlockPrivateNetworks = false`
  or an `AllowedHosts` entry for local dev.
