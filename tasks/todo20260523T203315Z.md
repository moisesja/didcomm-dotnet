**Assembly layout shipped**
- `DidComm.Core` (existing) — promoted the message-shape and `Jwk` types to public; added facade, resolution, secrets, and rotation namespaces.
- `DidComm.Extensions.DependencyInjection` — new; `services.AddDidComm(b => …)`.
- `DidComm.Adapters.NetDid` — new; `IKeyStore` → `ISecretsResolver` bridge (FR-SEC-04, with documented scope limit).
- `DidComm.TestSupport` — new; `InMemorySecretsResolver` outside `Core` to honour DD-02.

**Design decisions that diverged from the approved plan, with rationale**
1. **Sync-over-async at the seam, not pre-resolve.** The plan called for snapshotting outer-layer kids via `JweParser.PeekRecipients` and feeding a snapshot lookup into `EnvelopeReader.Unpack`. Implemented instead as `SyncSecretsAdapter` (a sync-over-async wrapper). Rationale: `EnvelopeReader` already calls `FindPresent` once + `TryGet` per matched recipient, so the snapshot would not have reduced I/O for the common single-layer case; nested envelopes would have required the same fallback anyway. The peek helper is shipped and documented for future caching work.
2. **Appendix B scope.** Vendored alice + bob only; deferred charlie / mediator1 / mediator2 to Phase 4 alongside the FR-ROUTE-* work that actually exercises them.
3. **Rotation L-005 mitigation.** No external `from_prior` KAT exists in the DIDComm spec, and reference impls build claims programmatically. Replaced "vendor a KAT" with five mutation tests (signature, sub, signer authorization, alg-curve match, structural validity) so the validator catches each individual bug class without depending on the builder for ground truth.
4. **Multibase verification methods skipped silently.** When `NetDidKeyService` encounters a method with only `publicKeyMultibase` (no `publicKeyJwk`), it skips the entry rather than throwing. Phase 3 only needs JWK-side resolution; multibase handling lands when a Phase 4+ fixture requires it.

**Open items / known limitations recorded for follow-up**
- `MessageTypeUri.IsValid` accepts a UUID-v4 default `id` even though FR-MSG-02 says "≤ 32 bytes". Documented Phase 2 carry-over; no change in Phase 3.
- `NetDidKeyStoreSecretsResolver` cannot surface decryption-path private keys (IKeyStore intentionally hides them). Documented in the class XML doc; consumer guidance is to pair it with an extractable store for the keys that need ECDH.
- `JweParser.PeekRecipients` is wired but not consumed today. Kept available for a later optimization pass.
