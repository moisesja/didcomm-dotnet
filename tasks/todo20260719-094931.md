# Issue #49 — Discover Features initiator-side round-trip · release prep v1.3.0

## Verdict on the issue

**Valid.** Every claim verified against source at `DidCommVersion` 1.2.0:

- `DiscoverFeaturesHandler.HandleAsync` drops any non-`queries` inbound (`DiscoverFeaturesHandler.cs:47-48`), so an inbound `disclose` is consumed and never surfaced.
- The registry is one-handler-per-PIURI, replace-on-register (`ProtocolHandlerRegistry.cs:33`), and the dispatcher resolves exactly one handler (`ProtocolDispatcher.cs:105`) — no observer/listener/middleware seam exists anywhere in `src` (grep confirms 0 matches).
- Construction/parse helpers exist (`CreateQuery`, `ReadDisclosures`) but there is no correlation-and-await API, so the requester role of Discover Features 2.0 is effectively unimplemented.
- Companion issue #50 (same root cause, same author) documents the general gap and marks a **general inbound-observer seam as the preferred fix**, explicitly noting it resolves both issues.

## Design decision

Implement the **general observer seam** (issue #50's preferred option 1) and build the
Discover Features initiator API on top of it, rather than a protocol-local pending-query
registry inside `DiscoverFeaturesHandler`:

- One mechanism serves both open issues (#49 and #50) — no second seam needed later.
- Read-only side channel: reply semantics, the FR-THR-04 loop guard, and the responder
  side of Discover Features are untouched.
- Purely additive public API; no wire change. Warrants a **minor** bump → **1.3.0**.

## Plan

### 1. Observer seam (closes #50's core ask) — hardened design

**Threat model (user-raised: "are we introducing a MITM sniffer?").** Observers are NOT
on the wire and NOT between parties: they run in-process on the recipient's side, are
registered only by the host application at DI-composition time, and see a message only
after the recipient's own keys unpacked it. Envelope crypto is untouched; mediators (the
actual untrusted middle parties in DIDComm routing) still see only encrypted forwards.
The seam sits inside the existing in-process trust boundary, which ALREADY contains a
strictly stronger primitive: `AddProtocol<T>()` + replace-on-register
(`ProtocolHandlerRegistry.cs:33`) lets any DI-composition code intercept AND alter/suppress
replies for any PIURI today. Residual risks to mitigate anyway (defense in depth):
(a) `Message` is mutable (`Body { get; set; }`, mutable attachments) — a naive observer
could tamper, not just observe; (b) `ProtocolContext` over-grants (carries the facade +
mutable thread store); (c) a stowaway observer from a compromised dependency should be
visible to operators.

- [ ] New `src/DidComm.Core/Protocols/IProtocolObserver.cs`:
      `string? ProtocolUriFilter { get; }` (least privilege: a PIURI, matched with the
      registry's same-protocol/major rules; `null` = observe all) and
      `Task OnMessageReceivedAsync(InboundObservation observation, CancellationToken ct)`.
- [ ] New `InboundObservation` record — the **narrow, read-only payload**: a defensive
      **deep clone** of the message (Body/attachments cloned once per inbound when ≥1
      observer matches — mutation by an observer can never reach handlers, replies, or
      the wire) + envelope-auth metadata copied from `UnpackResult` (`Encrypted`,
      `Authenticated`, `NonRepudiation`, `AnonymousSender`, `SenderKid`, `SignerKid`).
      Deliberately EXCLUDED: `DidCommClient` facade, `IThreadStateStore`, `ThreadState` —
      observers cannot send as the agent or mutate protocol state through the seam.
- [ ] `ProtocolDispatcher`: optional ctor param `IReadOnlyList<IProtocolObserver>? observers`.
      Notify **after** the dispatch outcome is determined (in all paths: handled, `NoReply`,
      `NoHandler`, ack-loop drops) — exactly once per inbound — so observer timing/state can
      never influence handling. Sequential await, per-observer try/catch + `LogWarning`;
      zero allocation/behavior change when no observer is registered.
- [ ] Startup audit trail: when the dispatcher (or builder) wires observers, log one
      Information line enumerating the registered observer types and their filters, so a
      stowaway registration is operator-visible.
- [ ] Registration is startup-only: observers arrive via ctor injection as an immutable
      list; no runtime add/remove API on the dispatcher.

### 2. Initiator API (closes #49)
- [ ] New `src/DidComm.Core/Protocols/DiscoverFeatures/DiscoverFeaturesClient.cs` —
      public sealed class implementing `IProtocolObserver`:
      ```csharp
      Task<IReadOnlyList<FeatureDisclosure>> QueryFeaturesAsync(
          string from, string to, IReadOnlyList<FeatureQuery> queries,
          TimeSpan timeout, Uri? serviceEndpointOverride = null, CancellationToken ct = default)
      ```
      Flow: `DiscoverFeatures.CreateQuery` → register `TaskCompletionSource` keyed by the
      query message id in a `ConcurrentDictionary` (**before** sending, to avoid the race) →
      `DidCommClient.SendAsync` (authcrypt: `From = from`) → await with timeout
      (`TimeoutException`) / cancellation; pending entry always removed in `finally`.
- [ ] Observer side (`ProtocolUriFilter = discover-features/2.0` — sees nothing else):
      complete the waiter when an inbound has `Type == DiscloseType` (case-insensitive)
      and `Thid` matches a pending query — **with anti-spoof checks**:
      - `message.From` must equal the queried `to` DID;
      - the envelope must authenticate the sender (`observation.Authenticated` — authcrypt
        or signed); an anoncrypt/plaintext `disclose` with a guessed `thid` MUST NOT
        complete a waiter (logged, ignored).
      Unmatched/unsolicited `disclose` stays terminal-leaf (current behavior preserved).
